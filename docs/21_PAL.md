# PAL: Program-aided Language Models

## 层 1：电梯演讲

**一句话概括**：PAL 提出让 LLM 生成代码作为推理步骤，但将计算任务交给 Python 解释器执行，在 13 个数学和符号推理任务上超越 CoT，GSM8K 基准上比 PaLM-540B+CoT 提升 15%，在大数字计算上提升 40%。

---

## 层 2：故事摘要

### 核心问题
LLM 擅长分解问题（如 CoT），但**计算能力差**：
- 能做简单算术：5 + 3 = 8 ✓
- 复杂计算就出错：123 × 456 = ? ✗
- 大数字处理更差：超出训练分布就失败

### 核心洞察
CMU 团队在 2022 年观察到：
1. LLM 学过自然语言 + 编程语言
2. LLM 能理解问题并分解为步骤
3. **为什么不让 LLM 写代码，让解释器执行计算？**

### PAL 框架大图

```
┌─────────────────────────────────────────────────────────┐
│              PAL vs CoT 对比                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  CoT (Chain-of-Thought):                                │
│  ┌─────────────────────────────────────────────────┐   │
│  │ Q: 23 元买 5 个 3 元的面包，剩多少钱？               │   │
│  │ A: 23 - 5×3 = 23 - 15 = 8 元                      │   │
│  │              ↑                                  │   │
│  │         LLM 自己计算（可能出错）                   │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  PAL (Program-Aided Language Model):                   │
│  ┌─────────────────────────────────────────────────┐   │
│  │ Q: 23 元买 5 个 3 元的面包，剩多少钱？               │   │
│  │ A: money = 23                                   │   │
│  │     cost_per_bread = 3                          │   │
│  │     num_breads = 5                              │   │
│  │     total_cost = num_breads * cost_per_bread    │   │
│  │     remaining = money - total_cost              │   │
│  │     print(remaining)  → 8                        │   │
│  │              ↑                                  │   │
│  │         Python 解释器计算（不会错）                │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 关键结果
- **GSM8K**: PAL 85.3% vs CoT 70.3% (+15%)
- **GSM-HARD** (大数字): PAL 68.5% vs CoT 28.5% (+40%)
- **13 个任务**: PAL 全面超越 CoT

## 层 3：深度精读

### 开场：一个典型的 CoT 失败场景

**时间**：2022 年
**地点**：CMU 实验室
**人物**：Luyu Gao, Aman Madaan, Shuyan Zhou 团队

**任务**：
```
Q: 面包师早上烤了 200 个面包，上午卖了 93 个，
   下午卖了 39 个，超市退回 6 个未售出的。
   还剩多少个面包？
```

**CoT 的输出**：
```
A: 200 - 93 - 39 + 6 = 62
   （实际计算：200-93=107, 107-39=68, 68+6=74）
   ❌ 计算错误！
```

**问题**：
- 问题分解是对的（知道要减法 + 加法）
- 但算术算错了
- 这种错误在复杂计算中很常见

"LLM 能理解问题，但计算不可靠，" 团队讨论，"为什么不让它写代码，让计算机做计算？"

这就是 PAL 的起点。

---

### 第一章：LLM 的计算局限

#### CoT 的成功与失败

**CoT 成功的情况**：
```
Q: 小明有 5 个苹果，又买了 3 个，现在几个？
CoT: 5 + 3 = 8 ✓
```

**CoT 失败的情况**：
```
Q: 2371 × 4829 = ?
CoT: 2371 × 4829 = 11449559（错误）
实际：11449559 → 11449859 ✓
```

**失败原因分析**：
1. **训练分布外**：大数字在训练数据中少见
2. **概率生成**：LLM 是预测下一个 token，不是真正计算
3. **误差累积**：多步计算中，一步错步步错

#### 关键洞察

团队观察到：
```
LLM 的能力 = 理解 + 分解 ✓
LLM 的弱点 = 精确计算 ✗

外部工具的能力 = 精确计算 ✓

组合起来 = LLM 理解分解 + 工具精确计算
```

**类比**：
```
人类数学家 vs 计算器：
- 数学家理解问题，写公式
- 计算器执行计算

PAL 就是让 LLM 当"数学家"，Python 当"计算器"
```

---

### 第二章：PAL 方法

#### PAL 的核心思想

**从 CoT 到 PAL 的演化**：

**CoT**：
```
Q → [LLM: 思考步骤] → 答案
    全部由 LLM 生成（包括计算）
```

**PAL**：
```
Q → [LLM: 生成代码] → [Python: 执行代码] → 答案
    LLM 只写代码         解释器计算
```

#### 实例对比

**例 1：简单算术**
```
Q: Olivia 有 23 元，买了 5 个 3 元的面包，剩多少钱？

CoT:
23 - 5×3 = 23 - 15 = 8 元

PAL:
money_initial = 23
bagels = 5
bagel_cost = 3
money_spent = bagels * bagel_cost
money_left = money_initial - money_spent
print(money_left)  # 输出：8
```

**例 2：多步推理**
```
Q: 罗杰有 5 个网球，又买了 2 罐，每罐 3 个球，现在几个？

PAL 代码：
# Roger started with 5 tennis balls
tennis_balls = 5

# 2 cans of 3 tennis balls each
bought_balls = 2 * 3

# Total tennis balls
answer = tennis_balls + bought_balls
print(answer)  # 输出：11
```

**例 3：带变量的问题**
```
Q: 一个数列 a1=1, an=an-1+2, 求 a10

PAL 代码：
a = [0] * 11
a[1] = 1
for i in range(2, 11):
    a[i] = a[i-1] + 2
answer = a[10]
print(answer)  # 输出：19
```

### 第三章：关键技术细节

#### Prompt 设计

**PAL Prompt 结构**：
```python
# Few-shot examples
Q: Olivia has $23...
A: money_initial = 23
   bagels = 5
   ...

# Test question
Q: [新问题]
A: [LLM 生成代码]
```

**关键设计原则**：
1. **有意义的变量名**：
   ```python
   # 好：
   money_initial = 23

   # 坏：
   x = 23
   ```

2. **注释说明**：
   ```python
   # Roger started with 5 tennis balls
   tennis_balls = 5
   ```

3. **自然语言 + 代码混合**：
   ```python
   # 2 cans of 3 tennis balls each is
   bought_balls = 2 * 3
   # tennis balls. The answer is
   answer = tennis_balls + bought_balls
   ```

#### 执行方式

**方式 1：一次性执行**
```
1. LLM 生成完整代码
2. 一次性执行所有代码
3. 输出结果
```

**方式 2：增量执行**
```
1. LLM 生成部分代码
2. 执行，获取中间结果
3. LLM 根据结果继续生成
4. 重复直到完成
```

论文使用方式 1（更简单）。

### 第五章：预期 vs 实际

### 你的直觉 vs PAL 的发现

| 维度 | 你的直觉/预期 | PAL 实际发现 | 为什么 |
|------|--------------|-------------|--------|
| **需要多少示例？** | 很多 | 3-6 个就够了 | 代码结构简单，容易学习 |
| **变量名重要吗？** | 不太重要 | 非常关键 | 有意义的变量名帮助 LLM 理解问题 |
| **代码能有多复杂？** | 只能简单代码 | 能生成循环、条件 | Codex 学过编程 |
| **执行错误怎么办？** | 很严重 | 少见且可处理 | LLM 通常生成有效代码 |

### 反直觉挑战

**问题：如果 LLM 生成的代码有 bug，会怎样？**

[先想 1 分钟...]

**直觉**："那结果就错了，整个方法就失败了"

**实际**：
- 代码通常是正确的（90%+ 可执行）
- 原因：LLM 学过大量代码
- 即使有错，也常是逻辑错，不是语法错

**PAL 的优势**：
> 把"计算错误"转化为"代码生成错误"
> 前者 LLM 无法控制，后者可以通过训练改进

### 第六章：局限性与改进

#### PAL 的局限

1. **依赖代码能力**：
   - 需要 LLM 学过编程
   - 小模型效果差

2. **安全问题**：
   - 执行生成的代码有风险
   - 需要沙箱环境

3. **调试困难**：
   - 代码出错难以定位
   - LLM 不知道错在哪

#### 改进方向

1. **更好的变量命名**：
   ```python
   # 自动优化变量名
   x123 → money_initial
   ```

2. **增量执行 + 反馈**：
   ```
   生成代码 → 执行 → 出错 → 反馈给 LLM → 修复
   ```

3. **多语言支持**：
   ```
   不仅 Python，还可以是：
   - SQL（数据库查询）
   - Mathematica（符号计算）
   - JavaScript（网页交互）
   ```
