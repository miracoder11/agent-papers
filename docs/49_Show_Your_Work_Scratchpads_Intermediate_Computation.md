## 层 1：电梯演讲

**一句话概括**：Google Research 在 2021 年提出 Scratchpad 方法，通过让语言模型输出中间计算步骤（类似于草稿纸），显著提升了模型执行多步计算和代码追踪的能力，为后续 Chain-of-Thought 等方法奠定了基础。

#### 1.2 为什么 Transformer 做多步计算困难

**计算资源限制**

```
Transformer 的计算特性：
- 固定层数（如 12 层、24 层）
- 固定计算量
- 无法根据任务难度调整

问题：
- 简单任务：可能够用
- 复杂任务：计算量不够
- 无法"慢慢想"
```

**与人类对比**

```
人类做计算：
- 可以在纸上写下来（外部存储）
- 一步一步算（多步推理）
- 算错了可以检查（自我验证）

Transformer：
- 所有计算在内部（隐式）
- 必须一步到位（单步预测）
- 无法检查（无自我验证）
```

**核心洞察**

> "如果让模型把中间步骤'写出来'呢？"
>
> "就像人类用草稿纸一样！"
>
> "这样模型就可以：
> 1. 分步计算
> 2. 利用前面的结果
> 3. 自我验证"

这就是 Scratchpad！

### 第二章：Scratchpad 方法

#### 2.1 核心思想

**Scratchpad 是什么**

```
Scratchpad = 草稿纸

人类用草稿纸：
- 写下中间计算步骤
- 一步一步推导
- 最后得出答案

模型用 Scratchpad：
- 输出中间 token 序列
- 每一步基于前面的步骤
- 最后输出答案
```

**形式化定义**

```
传统方法：
P(y | x)
直接预测答案 y

Scratchpad 方法：
P(s1, s2, ..., sk, y | x)
先输出中间步骤 s1, s2, ..., sk
最后输出答案 y

关键：
- 中间步骤是显式的文本
- 每一步可以依赖前面的步骤
- 训练时提供步骤监督
```

#### 2.2 任务 1：长加法

**问题设置**

```
输入：两个多位数
输出：它们的和

挑战：
- 需要逐位相加
- 需要处理进位
- 数字越长，步骤越多
```

**Scratchpad 表示**

```
例子：12345 + 67890

输入：
  12345
+ 67890
------

Scratchpad 输出：
  12345
+ 67890
------
  5 (5+0=5)
  3 (4+9=13, 写 3 进 1)
  2 (3+8+1=12, 写 2 进 1)
  0 (2+7+1=10, 写 0 进 1)
  8 (1+6+1=8)
------
  80235
```

**训练数据构造**

```python
# 生成训练样本
def generate_addition_data(num_digits):
    # 随机生成两个 num_digits 位的数
    a = random.randint(10**(num_digits-1), 10**num_digits - 1)
    b = random.randint(10**(num_digits-1), 10**num_digits - 1)

    # 计算正确答案
    result = a + b

    # 生成 Scratchpad 步骤
    scratchpad = generate_scratchpad(a, b, result)

    return {
        'input': f"{a} + {b} =",
        'scratchpad': scratchpad,
        'output': str(result)
    }

def generate_scratchpad(a, b, result):
    # 模拟手工加法，生成逐位计算过程
    steps = []
    carry = 0
    a_str = str(a).zfill(10)
    b_str = str(b).zfill(10)

    for i in reversed(range(len(a_str))):
        digit_a = int(a_str[i])
        digit_b = int(b_str[i])
        total = digit_a + digit_b + carry
        steps.append(f"{digit_a}+{digit_b}+{carry}={total}")
        carry = total // 10

    return '\n'.join(steps)
```

**实验结果**

```
长加法结果（泛化到更长数字）：
┌─────────────────────────────────────────┐
│ 数字长度 | 直接预测 | Scratchpad        │
├─────────────────────────────────────────┤
│ 5 位      | 45%      | 95%               │
│ 10 位     | 5%       | 92%               │
│ 20 位     | 0%       | 88%               │
│ 50 位     | 0%       | 75%               │
└─────────────────────────────────────────┘

关键发现：
- Scratchpad 显著提升性能
- 泛化能力更强（训练短数字，测试长数字）
```

#### 2.3 任务 2：多项式求值

**问题设置**

```
输入：多项式系数 + x 值
输出：多项式的值

例子：
p(x) = 3x³ + 2x² + x + 5
求 p(2) = ?

需要多步计算：
- 3 × 2³ = 24
- 2 × 2² = 8
- 1 × 2 = 2
- 24 + 8 + 2 + 5 = 39
```

**Scratchpad 表示**

```
输入：
p(x) = 3x³ + 2x² + x + 5
x = 2

Scratchpad 输出：
x³ = 8
x² = 4
3x³ = 3 × 8 = 24
2x² = 2 × 4 = 8
x = 2
常数项 = 5
总和 = 24 + 8 + 2 + 5 = 39

输出：39
```

**结果**

```
多项式求值结果：
┌─────────────────────────────────────────┐
│ 设置        | 直接预测 | Scratchpad     │
├─────────────────────────────────────────┤
│ Fine-tune | 35%      | 78%              │
│ Few-shot  | 28%      | 65%              │
└─────────────────────────────────────────┘
```

#### 2.4 任务 3：代码执行追踪

**问题设置**

```
输入：一段代码
输出：执行结果或变量状态

挑战：
- 需要理解循环
- 需要追踪变量变化
- 需要条件分支推理
```

**Scratchpad 表示**

```python
代码：
def remove_Occ(s, ch):
    for i in range(len(s)):
        if s[i] == ch:
            s = s[0:i] + s[i+1:]
            break
    return s

output = remove_Occ("PHP", "P")

Scratchpad 输出（逐行追踪）：
line: def remove_Occ(s, ch):
state: {}

line: output = remove_Occ("PHP", "P")
state: {"remove_Occ": <function>}

# 进入函数
line:     for i in range(len(s)):
state: {"s": "PHP", "ch": "P"}

line:         if s[i] == ch:
state: {"s": "PHP", "ch": "P", "i": 0}
# s[0] = 'P' == 'P'，条件为真

line:             s = s[0:i] + s[i+1:]
state: {"s": "HP", "ch": "P", "i": 0}

line:             break
state: {"s": "HP", "ch": "P", "i": 0}

line:     return s
state: {"output": "HP"}

输出："HP"
```

**结果**

```
代码执行预测结果：
┌─────────────────────────────────────────┐
│ 任务              | 直接 | Scratchpad   │
├─────────────────────────────────────────┤
│ 简单赋值         | 65%  | 85%           │
│ 条件语句         | 45%  | 78%           │
│ 循环（少次）     | 30%  | 72%           │
│ 循环（多次）     | 10%  | 55%           │
└─────────────────────────────────────────┘

结论：
- Scratchpad 显著提升代码理解
- 循环越多，提升越大
```

### 第三章：核心概念 - 大量实例

#### 3.1 为什么 Scratchpad 有效

**生活类比 1：心算 vs 笔算**

```
心算（直接预测）：
- 12345 × 67890 = ?
- 很难，容易出错
- 需要记住所有中间结果

笔算（Scratchpad）：
- 在纸上写下来
- 一步一步算
- 可以检查

Scratchpad 就是让模型"笔算"！
```

**生活类比 2：走迷宫**

```
直接预测：
- 站在迷宫入口
- 直接猜出口在哪里
- 几乎不可能对

Scratchpad：
- 一边走一边标记
- 记录走过的路
- 最终找到出口

多步计算就像走迷宫！
```

**计算图类比**

```
直接预测：
输入 → [黑箱] → 输出

Scratchpad：
输入 → 步骤 1 → 步骤 2 → ... → 步骤 n → 输出
       ↓         ↓              ↓
     中间结果 1  中间结果 2    中间结果 n

关键区别：
- 中间结果是显式的
- 每一步可以依赖前面的结果
- 训练时可以提供监督
```

#### 3.2 泛化能力

**什么是泛化**

```
训练：短数字加法（5-10 位）
测试：长数字加法（20-50 位）

为什么能泛化？
- 学会了加法算法
- 而不是记住答案
- 步骤可以任意延长
```

**类比：学数学**

```
死记硬背：
- 记住 23+45=68
- 但不会算 234+456

学会方法：
- 学会竖式加法
- 任何位数都会算

Scratchpad 让模型学会方法！
```

#### 3.3 训练方式

**监督微调**

```python
# 训练数据
training_data = [
    {
        'input': '123 + 456 =',
        'scratchpad': '123\n+456\n---\n3 (3+6=9)\n7 (2+5=7)\n5 (1+4=5)\n---\n579',
        'output': '579'
    },
    # ... 更多样本
]

# 训练目标：最大化对数似然
loss = -log P(scratchpad, output | input)

# 反向传播更新模型
```

**Few-shot 设置**

```
测试时提供示例：

示例 1：
输入：23 + 45 =
Scratchpad: 23
            +45
            ---
            8 (3+5=8)
            6 (2+4=6)
            ---
            68
输出：68

示例 2：
输入：123 + 456 =
Scratchpad: ...
输出：579

测试：
输入：12345 + 67890 =
Scratchpad: ...（模型自己生成）
输出：80235
```

### 第四章：预期 vs 实际

#### 4.1 直觉预测 vs 实际结果

**预期 1：更大的模型 = 更好的计算**

| 你的直觉 | 实际 | 为什么 |
|---------|------|--------|
| GPT-3 应该会加法 | 多位数就不会 | 不是真计算，是模式匹配 |
| 微调后就会了 | 泛化还是差 | 没学会算法本质 |
| Scratchpad 帮助有限 | 提升巨大 | 改变了计算方式 |

**预期 2：Scratchpad 只是形式**

| 你的直觉 | 实际 | 为什么 |
|---------|------|--------|
| 答案对就行 | 步骤很重要 | 步骤强制正确推理 |
| 模型知道怎么做 | 需要显式输出 | 隐式知识不够用 |
| 训练简单任务就会 | 需要多步监督 | 学会的是方法 |

**预期 3：只能做数学**

| 你的直觉 | 实际 | 为什么 |
|---------|------|--------|
| 只适合算术 | 适合所有多步任务 | 本质是分步推理 |
| 代码不行 | 代码追踪很好 | 代码是天然的步骤 |
| 推理不行 | 推理有提升 | 推理需要中间步骤 |

#### 4.2 反直觉挑战

**问题：如果模型知道答案，为什么还需要中间步骤？**

[先想 1 分钟...]

**直觉**：知道答案就够了，步骤是多余的。

**实际**：模型不是真"知道"，是"猜"。

```
解释：
- 大模型的回答是基于模式匹配
- 看到"123+456"，匹配到"579"
- 但不知道 579 怎么来的

遇到新情况：
- "12345+67890" 没见过的模式
- 就不会了

Scratchpad 的作用：
- 强制模型学习算法
- 而不是记忆答案
- 所以能泛化
```

### 第六章：与其他方法对比

#### 6.1 方法对比

| 方法 | 中间步骤 | 监督需求 | 泛化能力 | 适用任务 |
|------|----------|----------|----------|----------|
| 直接预测 | ❌ | 仅需答案 | 差 | 简单任务 |
| Scratchpad | ✅ | 需要步骤 | 好 | 多步计算 |
| Chain-of-Thought | ✅ | Few-shot | 好 | 推理任务 |
| Program Synthesis | ✅ | 需要代码 | 最好 | 算法任务 |

#### 6.2 局限性

```
1. 需要步骤监督
   - 训练数据要包含中间步骤
   - 构造成本高

2. 步骤格式固定
   - 只能做定义好的算法
   - 无法灵活变通

3. 错误传播
   - 一步错，后面都错
   - 没有纠错机制

4. 计算效率
   - 输出更长序列
   - 推理时间增加
```

#### 7.2 实现示例

```python
# 使用 Scratchpad 进行加法
from transformers import GPT2LMHeadModel, GPT2Tokenizer

model = GPT2LMHeadModel.from_pretrained('your-finetuned-model')
tokenizer = GPT2Tokenizer.from_pretrained('your-finetuned-model')

def add_with_scratchpad(a, b):
    input_text = f"{a} + {b} =\n"
    inputs = tokenizer(input_text, return_tensors='pt')

    # 生成 Scratchpad 和答案
    outputs = model.generate(
        **inputs,
        max_length=200,
        num_return_sequences=1
    )

    result = tokenizer.decode(outputs[0])
    return result

# 使用
print(add_with_scratchpad(12345, 67890))
# 输出包含中间步骤和最终答案
```

### 总结

**Scratchpad 的核心贡献**：

1. **方法创新**
   - 首次提出让模型输出中间步骤
   - 不改变架构，只改变任务设计

2. **性能提升**
   - 多步计算任务显著提升
   - 泛化能力增强

3. **影响力**
   - 启发了 Chain-of-Thought
   - 开启了"思考过程显式化"的研究方向

**一句话总结**：
> Scratchpad 通过让语言模型输出中间计算步骤，就像人类用草稿纸一样，显著提升了模型执行多步计算和代码理解的能力，是推理增强领域的重要里程碑。
