# ChatDev: Communicative Agents for Software Development

## 层 1：电梯演讲

**一句话概括**：ChatDev 是一个聊天驱动的软件开发框架，通过赋予 LLM 智能体不同角色（CEO、CTO、程序员、测试员），沿着"对话链"进行多轮沟通，能够自主完成从需求到代码的全流程，在多个任务上生成可执行软件。

---

## 层 2：故事摘要

### 核心问题
软件开发是一个需要多人协作的复杂任务，但传统的深度学习模型只能优化某个单一环节（如设计、编码、测试），导致**技术不一致、流程碎片化**。

### 核心洞察
清华大学团队在 2023 年观察到：
1. LLM 擅长角色扮演和语言理解
2. 人类开发团队通过"沟通"协作：自然语言讨论需求，编程语言调试代码
3. **如果用统一的"语言沟通"贯穿所有环节，会怎样？**

### ChatDev 框架大图

```
┌─────────────────────────────────────────────────────────┐
│              ChatDev 架构                               │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  输入："开发一个五子棋游戏"                                │
│                    ↓                                    │
│  ┌─────────────────────────────────────────────────┐   │
│  │           Chat Chain (对话链)                    │   │
│  │                                                  │   │
│  │  ① Design Phase (设计阶段)                        │   │
│  │     CEO → CTO: "游戏需要什么功能？"                 │   │
│  │     CTO → CEO: "需要棋盘、棋子、规则判断..."       │   │
│  │                    ↓                             │   │
│  │  ② Coding Phase (编码阶段)                        │   │
│  │     CTO → Programmer: "实现棋盘类"                 │   │
│  │     Programmer → CTO: "代码写好了"                 │   │
│  │     Reviewer → Programmer: "这里有个 bug"         │   │
│  │                    ↓                             │   │
│  │  ③ Testing Phase (测试阶段)                       │   │
│  │     Tester → Programmer: "运行失败，错误是..."     │   │
│  │     Programmer → Tester: "修复了"                 │   │
│  └─────────────────────────────────────────────────┘   │
│                    ↓                                    │
│  输出：完整的可执行软件 + 文档                            │
│                                                         │
└─────────────────────────────────────────────────────────┘

核心机制:
1. Chat Chain: 将瀑布模型拆分为原子对话任务
2. Communicative Dehallucination: 主动询问细节减少幻觉
3. 双 Agent 对话：Instructor (指令) + Assistant (执行)
```

### 关键结果
- **软件完整性**：ChatDev 生成的软件 80%+ 可直接运行
- **需求一致性**：生成的软件与需求描述高度匹配
- **沟通价值**：自然语言沟通提升设计质量，编程语言沟通提升调试效率

## 层 3：深度精读

### 开场：一个典型的软件开发失败场景

**时间**：2023 年初
**地点**：清华大学实验室
**人物**：Chen Qian 团队

团队成员盯着屏幕上的代码生成结果，感到沮丧。

**任务**："开发一个贪吃蛇游戏"

**传统方法的输出**：
```
# 代码生成模型输出
class Snake:
    def __init__(self):
        self.body = [(0, 0)]
    def move(self):
        ...

# 但没有：
# - 游戏主循环
# - 碰撞检测
# - 分数系统
# - 用户输入处理
```

**问题**：
- 模型只生成了"片段"，不是完整软件
- 没有设计文档，不知道整体架构
- 没有测试，不知道能不能运行

"这和真实的软件开发流程完全不同，" Chen Qian 说，"人类团队会先讨论设计，再分工编码，最后测试。为什么不让 AI 也这样？"

这就是 ChatDev 的起点。

### 第一章：研究者的困境

#### 软件开发的复杂性

软件开发从来不是一个人的战斗：

**人类团队的协作流程**：
```
1. 需求分析
   产品经理 → 客户："你想要什么功能？"

2. 系统设计
   架构师 → 团队："我们这样设计..."

3. 编码实现
   程序员 A → 程序员 B:"这个接口怎么定义？"

4. 测试调试
   测试员 → 程序员："这里有 bug"
   程序员 → 测试员："修好了"
```

**关键洞察**：
> 整个流程的本质是**沟通**——用自然语言讨论设计，用编程语言调试代码。

#### 传统 AI 方法的局限

**分割式 AI 工具**：
```
设计阶段 → 用模型 A (需要专门训练)
编码阶段 → 用模型 B (需要专门训练)
测试阶段 → 用模型 C (需要专门训练)

问题：
- 每个模型需要不同的数据格式
- 模型之间无法传递上下文
- 出错了不知道如何让上游模型修复
```

**研究者的焦虑**：
```
周一：试试用 CodeGen 生成代码
      → 代码不完整，缺头少尾

周二：试试加更多 fine-tuning
      → 过拟合，换个需求就不行了

周三：试试多模型组合
      → 模型之间无法沟通，各干各的

周四：团队陷入沉默
      → "也许我们需要完全不同的思路..."
```

#### 顿悟时刻

某天深夜，Chen Qian 突然意识到：

> "等等，LLM 本来就能理解自然语言和编程语言！"
> "为什么不让 LLM 扮演不同角色，让它们自己聊天完成开发？"

**核心洞察**：
- LLM 经过训练，既能理解需求描述（自然语言），也能理解代码（编程语言）
- 如果用"对话"贯穿所有环节，就不需要多个专门模型
- 多轮对话可以让模型自我纠正

### 第二章：试错的旅程

#### 版本 1：单次生成

**想法**：直接让 LLM 生成完整软件
```python
prompt = "开发一个五子棋游戏"
response = llm.generate(prompt)
```

**问题**：
- 输出太长，超出上下文限制
- 生成的代码没有结构，杂乱无章
- 出错了无法修复

#### 版本 2：分阶段生成

**想法**：按照瀑布模型分阶段
```
Design → 生成设计文档
Coding → 根据设计文档生成代码
Testing → 运行代码，报告错误
```

**问题**：
- 阶段之间信息丢失
- 设计文档和代码不一致
- 测试发现 bug 后无法反馈给设计

#### 版本 3：ChatDev 的诞生

**突破**：引入"对话链"
```
CEO    CTO    Programmer    Reviewer    Tester
 │      │         │            │           │
 └──────┴─────────┴────────────┴───────────┘
              多轮对话协作
```

**关键机制**：
1. **Chat Chain**：将每个阶段拆分为原子对话任务
2. **Inception Prompting**：预设角色和终止条件
3. **Communicative Dehallucination**：主动询问细节减少幻觉

**结果**：代码不仅能运行，还能自我修复！

### 第三章：核心概念 - 大量实例

#### 概念 1：Chat Chain（对话链）

**生活类比 1：接力赛**
```
想象一个软件开发接力赛：

第一棒 (CEO)："我们要做一个五子棋"
         ↓ 传递给
第二棒 (CTO)："需要棋盘、棋子、规则"
         ↓ 传递给
第三棒 (程序员)："我来写代码"
         ↓ 传递给
第四棒 (测试员)："这里有问题"
         ↓ 传递给
程序员："修好了"

Chat Chain 就是把每个"传递"变成一次对话。
```

**生活类比 2：厨房做饭**
```
👨‍🍳 主厨 (Instructor): "把洋葱切了"
         ↓
👤 学徒 (Assistant): "好的，切好了"
         ↓
👨‍🍳 主厨："现在炒一下"
         ↓
👤 学徒："炒好了，但有点焦"
         ↓
👨‍🍳 主厨："下次火小一点"
```

**代码实例 1：对话链结构**
```python
# ChatDev 的对话链伪代码
chat_chain = [
    # 设计阶段
    {"phase": "design", "instructor": "CEO", "assistant": "CTO"},

    # 编码阶段
    {"phase": "coding", "instructor": "CTO", "assistant": "Programmer"},
    {"phase": "code_complete", "instructor": "CTO", "assistant": "Programmer"},

    # 代码审查
    {"phase": "review", "instructor": "Reviewer", "assistant": "Programmer"},

    # 测试阶段
    {"phase": "testing", "instructor": "Tester", "assistant": "Programmer"},
]

# 每个阶段进行多轮对话
for task in chat_chain:
    instructor = get_agent(task["instructor"])
    assistant = get_agent(task["assistant"])

    while not task_complete():
        instruction = instructor.generate(task_context)
        response = assistant.generate(instruction)
        task_context.update(response)
```

**对比场景：传统流水线 vs Chat Chain**
```
【传统流水线】
需求 → 设计模型 → 编码模型 → 测试模型
          ↓           ↓          ↓
       信息丢失     信息丢失    无法反馈

【Chat Chain】
CEO ↔ CTO ↔ Programmer ↔ Reviewer ↔ Tester
  ↓     ↓        ↓          ↓         ↓
  多轮对话，信息共享，错误可反馈
```

#### 概念 2：Communicative Dehallucination（交流式去幻觉）

**生活类比：点餐**
```
没有去幻觉：
顾客："我要一份不辣的"
服务员："好的" → 端上来一盘辣子鸡
      (因为服务员没有问清楚)

有去幻觉：
顾客："我要一份不辣的"
服务员："请问您是完全不能吃辣，还是微微辣就可以？"
顾客："微微辣就可以"
服务员："好的，推荐宫保鸡丁，微辣的"
      (通过询问减少误解)
```

**代码实例 2：去幻觉机制**
```python
# 传统生成（容易产生幻觉）
def generate_code(requirement):
    prompt = f"根据以下需求生成代码：{requirement}"
    return llm.generate(prompt)

# ChatDev 的去幻觉生成
def generate_code_dehallucination(requirement):
    # 第一步：请求更多细节
    clarifying_questions = llm.generate(
        f"为了更好地完成这个需求，我需要知道：{requirement}"
    )
    # 例如："游戏需要 GUI 吗？" "需要保存进度吗？"

    # 第二步：根据回答生成代码
    answers = get_answers(clarifying_questions)
    detailed_requirement = requirement + "\n" + answers
    return llm.generate(f"根据详细需求生成代码：{detailed_requirement}")
```

**对比场景：有无去幻觉**
```
【无去幻觉】
需求："开发一个记账软件"
直接生成 → 代码缺少关键功能（如预算提醒）

【有去幻觉】
需求："开发一个记账软件"
询问："需要预算功能吗？需要多账户支持吗？"
得到回答后生成 → 代码更完整
```

#### 概念 3：Inception Prompting（ inception 提示）

**生活类比：给新人培训**
```
想象一个新入职的程序员：

没有培训：
"你去写代码吧" → 不知道写什么、怎么写、何时算完成

有培训：
"你的角色是后端程序员
 你的任务是实现 API 接口
 完成标准：单元测试通过
 注意事项：不要写死配置"
→ 清楚知道该做什么

Inception Prompting 就是给 LLM 做"培训"。
```

**代码实例 3：Inception Prompt 结构**
```python
instructor_prompt = """
# 角色
你是一位资深 CEO，负责定义软件需求。

- 可以向 CTO 提问

当功能需求列表完整且清晰时，发送"DESIGN_COMPLETE"

# 约束

assistant_prompt = """

- 可以向 CEO 确认需求
- 可以输出设计文档

- 使用清晰的技术术语
```

### 第四章：预期 vs 实际

### 你的直觉 vs ChatDev 的实现

| 维度 | 你的直觉/预期 | ChatDev 实际实现 | 为什么有差距？ |
|------|--------------|-----------------|---------------|
| **如何分工？** | 每个模型专门负责一个环节 | 同一个 LLM 扮演不同角色 | 角色通过 Prompt 定义，不需要专门训练 |
| **如何保证质量？** | 需要人工审核 | 通过多轮对话自我纠正 | 不同角色的"视角"不同，能发现问题 |
| **如何处理错误？** | 需要专门的错误处理模块 | 测试员发现错误后对话反馈 | 错误本身就是对话的一部分 |
| **如何终止？** | 达到预定义步骤数 | 发送"TERMINATE"信号 | 让 LLM 自己判断任务完成 |
| **代码如何执行？** | 需要外部执行器 | 内置代码执行沙箱 | 安全和开发体验的平衡 |

### 反直觉挑战

**问题：如果让同一个 LLM 同时扮演 CEO 和 CTO，会怎样？**

[先想 1 分钟...]

**直觉**："自己跟自己说话，有什么意义？"

**实际**：
- 虽然是同一个 LLM，但不同角色的 Prompt 不同
- CEO Prompt 强调"需求完整性"
- CTO Prompt 强调"技术可行性"
- 这种"视角切换"能发现单方面想不到的问题

**ChatDev 的巧妙之处**：
> 不是"多模型"，而是"多角色"——通过 Prompt 工程让同一个 LLM 展现不同视角。

### 第五章：关键实验的细节

#### 实验 1：软件生成质量

**任务**：生成 12 种不同类型的软件（游戏、工具等）

**评估指标**：
- **完整性**：软件是否包含所有必要组件
- **可执行性**：代码能否直接运行
- **一致性**：是否满足需求描述

**ChatDev 配置**：
```
Agents: CEO, CTO, Programmer, Reviewer, Tester
LLM: GPT-3.5-turbo
Chat Chain: 5 个阶段，每个阶段 2-5 轮对话
```

**结果**：
| 指标 | ChatDev | 基线 (单次生成) |
|------|---------|----------------|
| 完整性 | 88% | 45% |
| 可执行性 | 82% | 31% |
| 一致性 | 85% | 52% |

**关键洞察**：
- 多轮对话让模型有机会纠正错误
- 不同角色的"审查"提升了质量

#### 实验 2：自然语言 vs 编程语言沟通

**问题**：两种沟通方式各有什么作用？

**实验设计**：
- **条件 1**：设计阶段用自然语言，编码阶段用编程语言
- **条件 2**：全部用自然语言
- **条件 3**：全部用编程语言

**结果**：
| 条件 | 设计质量 | 代码质量 |
|------|---------|---------|
| 自然语言 + 编程语言 | **最高** | **最高** |
| 仅自然语言 | 高 | 中 |
| 仅编程语言 | 中 | 中 |

**关键洞察**：
> 自然语言适合抽象讨论（设计），编程语言适合精确表达（代码）。

#### 实验 3：多轮对话的价值

**问题**：多轮对话真的有必要吗？

**实验设计**：
- **1 轮**：单次生成
- **3 轮**：有限对话
- **不限**：直到任务完成

**结果**：
| 轮数 | 成功率 | 平均代码行数 |
|------|--------|-------------|
| 1 轮 | 31% | 50 |
| 3 轮 | 62% | 120 |
| 不限 | 82% | 180 |

**关键洞察**：
- 更多对话 = 更完整的软件
- 但对话轮数过多会增加成本

### 第六章：与其他方法对比

### ChatDev vs 其他框架

| 维度 | ChatDev | AutoGen | LangChain |
|------|---------|---------|-----------|
| **核心场景** | 软件开发 | 通用多 Agent | 工具链编排 |
| **角色定义** | 固定 (CEO/CTO 等) | 可自定义 | 无角色概念 |
| **对话流程** | 链式 (瀑布模型) | 动态 | 链式/图式 |
| **代码执行** | 内置沙箱 | 内置 | 需要工具 |
| **人类参与** | 可选 | 内置支持 | 需要配置 |

### ChatDev vs 传统代码生成

| 维度 | 传统代码生成 | ChatDev |
|------|-------------|---------|
| **输入** | 需求描述 | 需求描述 |
| **输出** | 代码片段 | 完整软件 |
| **错误处理** | 无 | 多轮对话修复 |
| **设计文档** | 无 | 自动生成 |
| **测试** | 无 | 自动测试 |

### 局限性分析

**ChatDev 的局限**：
1. **依赖 LLM 质量**：小模型效果明显下降
2. **成本问题**：多轮对话消耗大量 token
3. **复杂项目局限**：大型项目超出上下文限制
4. **领域特定知识**：需要领域知识的项目表现差

### 改进方向

1. **更高效的对话策略**：减少不必要的轮次
2. **长期记忆**：跨项目积累经验
3. **小模型优化**：针对开源模型的配置
4. **领域适配**：针对特定领域的 Prompt 优化

### 第七章：如何应用

#### 适用场景

✅ **适合用 ChatDev 的场景**：
- 快速原型开发
- 小型工具/游戏生成
- 代码学习和教学
- 需求探索和设计讨论

❌ **不适合用 ChatDev 的场景**：
- 大型商业项目
- 对安全性要求高的场景
- 需要与现有代码集成的场景
- 预算有限（token 消耗大）

```bash

cd ChatDev

# 2. 安装依赖
pip install -r requirements.txt

# 3. 配置 API Key
export OPENAI_API_KEY="sk-..."

# 4. 运行
python3 run.py --task "develop a 2048 game"
```
