# 从 30 行代码到自治团队：Claude Code 的演化之路

## 开场：一个简单的问题

语言模型能推理代码，但碰不到真实世界。

你可以问它"怎么读取一个文件"，它会告诉你用 `open()` 函数；但如果你问"这个项目的测试覆盖率是多少"，它就没辙了——因为它读不到文件，跑不了命令，看不到报错。

没有循环，每次工具调用你都得手动把结果粘回去。你自己就是那个循环。

这就是一切问题的起点。

---

## 起点：30 行代码

s01 给出了最简洁的答案：

```python
def agent_loop(query):
    messages = [{"role": "user", "content": query}]
    while True:
        response = client.messages.create(
            model=MODEL, system=SYSTEM, messages=messages,
            tools=TOOLS, max_tokens=8000,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return

        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        messages.append({"role": "user", "content": results})
```

**格言**："One loop & Bash is all you need"——一个工具 + 一个循环 = 一个智能体。

这就是整个智能体的核心。后面 11 个章节都在这个循环上叠加机制，循环本身始终不变。

这篇代码对应 2022 年的 **ReAct 论文**。论文的核心思想是：模型不应该只"想"，还应该"动"。推理和行动交织在一起，形成一个"思考-行动-观察"的循环。

---

## 演化第一站：加工具

只有 `bash` 时，所有操作都走 shell。`cat` 截断不可预测，`sed` 遇到特殊字符就崩，每次 bash 调用都是不受约束的安全面。

s02 的解决方案是 **dispatch map**：

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

**格言**："加一个工具，只加一个 handler"——循环不用动，新工具注册进 dispatch map 就行。

这里还有一个关键设计：**路径沙箱**。`safe_path()` 函数防止工具逃逸工作区，这是专用工具在安全面上的优势。

这对应 2023 年的 **Toolformer 论文**。论文的核心是：模型可以学会什么时候调用工具、传什么参数、怎么解析返回值。工具不再是"API 调用"，而是模型能力的一部分。

---

## 演化第二站：上下文隔离

智能体工作越久，messages 数组越胖。每次读文件、跑命令的输出都永久留在上下文里。"这个项目用什么测试框架？"可能要读 5 个文件，但父智能体只需要一个词："pytest"。

s04 的解决方案是 **Subagent**：

```
Parent agent                     Subagent
+------------------+             +------------------+
| messages=[...]   |             | messages=[]      | <-- fresh
|                  |  dispatch   |                  |
| tool: task       | ----------> | while tool_use:  |
|   prompt="..."   |             |   call tools     |
|                  |  summary    |   append results |
|   result = "..." | <---------- | return last text |
+------------------+             +------------------+
```

**格言**："大任务拆小，每个小任务干净的上下文"——子智能体用独立 messages[]，不污染主对话。

很多人误以为 Subagent 的价值是"并行加速"。其实不然。**Subagent 的核心价值是隔离**——让大量中间产物不要挤占主线程的有效上下文。扫代码库、跑测试、做审查，这些会产生大量输出的任务，交给 Subagent 做，主线程只拿一个摘要，干净很多。

子智能体可能跑了 30+ 次工具调用，但整个消息历史直接丢弃。父智能体收到的只是一段摘要文本。

---

## 演化第三站：按需知识

你希望智能体遵循特定领域的工作流：git 约定、测试模式、代码审查清单。全塞进系统提示太浪费——10 个技能，每个 2000 token，就是 20,000 token，大部分跟当前任务毫无关系。

s05 的解决方案是 **Skills 两层注入**：

```
System prompt (Layer 1 -- always present):
+--------------------------------------+
| You are a coding agent.              |
| Skills available:                    |
|   - git: Git workflow helpers        |  ~100 tokens/skill
|   - test: Testing best practices     |
+--------------------------------------+

When model calls load_skill("git"):
+--------------------------------------+
| tool_result (Layer 2 -- on demand):  |
| <skill name="git">                   |
|   Full git workflow instructions...  |  ~2000 tokens
| </skill>                             |
+--------------------------------------+
```

**格言**："用到什么知识，临时加载什么知识"——通过 tool_result 注入，不塞 system prompt。

第一层：系统提示中放技能名称（低成本）。第二层：tool_result 中按需放完整内容。

这背后是 **Prompt Caching 架构**的考量。缓存是按前缀匹配工作的，系统提示和工具定义放在前面，可以长期命中缓存。如果把所有技能内容都塞进系统提示，每次会话成本都会飙升。

## 演化第四站：上下文压缩

上下文窗口是有限的。读一个 1000 行的文件就吃掉约 4000 token；读 30 个文件、跑 20 条命令，轻松突破 100k token。

但问题比想象中更严重。

深度文章中有一个关键数据：**MCP 工具定义占 12.5% 上下文**。一个典型 MCP Server（如 GitHub）包含 20-30 个工具定义，每个约 200 tokens，合计 4,000-6,000 tokens。接 5 个 Server，光这部分固定开销就到了 25,000 tokens。

我第一次算出这个数字时，真没想到有这么多。

s06 的解决方案是 **三层压缩**：

```
[Layer 1: micro_compact]        (silent, every turn)
  Replace tool_result > 3 turns old with "[Previous: used {tool_name}]"
        |
        v
[Layer 2: auto_compact]
  token > 50000 → Save transcript, LLM summarizes, Replace all messages
        |
        v
[Layer 3: compact tool]
  Model calls compact explicitly.
```

**格言**："上下文总会满，要有办法腾地方"——三层压缩策略，换来无限会话。

还有一个细节：压缩后智能体可能忘了自己是谁。这就是 **身份重注入**：

```python
if len(messages) <= 3:
    messages.insert(0, {"role": "user",
        "content": f"<identity>You are '{name}', role: {role}...</identity>"})
```

压缩算法会丢掉历史，但身份必须保留。

## 演化第五站：团队协作

子智能体（s04）是一次性的：生成、干活、返回摘要、消亡。没有身份，没有跨调用的记忆。

真正的团队协作需要三样东西：(1) 能跨多轮对话存活的持久智能体，(2) 身份和生命周期管理，(3) 智能体之间的通信通道。

s09 的解决方案是 **Agent Teams + JSONL 邮箱**：

```
.team/
  config.json           <- team roster + statuses
  inbox/
    alice.jsonl         <- append-only, drain-on-read
    bob.jsonl
    lead.jsonl
```

**格言**："任务太大一个人干不完，要能分给队友"——持久化队友 + JSONL 邮箱。

每个队友在每次 LLM 调用前检查收件箱，将消息注入上下文。append-only 的 JSONL 文件充当邮箱，发送即追加，读取即清空。

---

## 终局：自治智能体

s09 中，队友只在被明确指派时才动。领导得给每个队友写 prompt，任务看板上 10 个未认领的任务得手动分配。这扩展不了。

s11 的解决方案是 **自治智能体**：

```
Teammate lifecycle with idle cycle:
+-------+
| spawn |
+---+---+
    |
    v
+-------+   tool_use     +-------+
| WORK  | <------------- |  LLM  |
+---+---+                +-------+
    |
    | stop_reason != tool_use
    v
+--------+
|  IDLE  |  poll every 5s for up to 60s
+---+----+
    |
    +---> check inbox --> message? ----------> WORK
    |
    +---> scan .tasks/ --> unclaimed? -------> claim -> WORK
    |
    +---> 60s timeout ----------------------> SHUTDOWN
```

**格言**："队友自己看看板，有活就认领"——不需要领导逐个分配，自组织。

队友自己扫描任务看板，认领没人做的任务，做完再找下一个。空闲阶段循环轮询收件箱和任务看板，找到工作就恢复工作状态，60 秒无事可做就自动关机。

这对应 2023 年的 **MetaGPT** 和 **AutoGen** 论文。论文的核心思想是：多智能体协作不是简单的"分工"，而是要有角色定义、通信协议和自治调度。

---

## 番外：论文地图

把机制和论文对应起来，会看到一个清晰的脉络：

| 机制 | 论文 | 核心思想 |
|------|------|----------|
| Agent Loop (s01) | ReAct (2022) | 推理+行动的循环 |
| 工具使用 (s02) | Toolformer (2023) | 模型学会调用工具 |
| 验证闭环 | Reflexion (2023) | 自我反思修正 |
| 多智能体协作 (s09-s11) | MetaGPT, AutoGen (2023) | 多角色协作 |

从论文到工程实践，Claude Code 的每个机制都能找到学术渊源。

---

## 扩展：未来可能

演化到 s11 就结束了吗？远远没有。

**真正的长期记忆**：不只是 200K，而是跨会话的记忆。今天讨论的架构决策，下周还能记住。

**跨会话学习**：每次用都变得更懂你。你的代码风格、项目约定、常见坑，它自己慢慢学会。

**多 Agent 知识共享**：队友之间共享学到的东西。Alice 发现了一个坑，Bob 自动知道避开。

**自我调试和自我修复**：出问题自己排查自己修。不需要人类介入，自己完成整个闭环。

**主动式常驻助手**：从"踹一下动一下"到"自己找活干"。不是你给它派任务，是它告诉你有什么该做。

---

## 尾声

回到最开始那 30 行代码。

它很简单，但已经包含了智能体的本质：一个循环、一个工具、一个退出条件。

后面的一切——dispatch map、Subagent、Skills、压缩、团队、自治——都是在这个基础上叠加机制。每个机制解决一个具体问题，每个问题又引出新的问题。

这不是终点，这只是开始。

当你理解了这套演化逻辑，你会发现：Claude Code 不是魔法，而是一个不断解决问题的工程系统。每个设计决策背后，都有一个具体的问题和一个务实的答案。

理解这些，你就能从"使用者"进阶为"设计者"。不再只是用它，而是理解它、驾驭它、改进它。
