# Claude Code 六层架构解析：从 Agent Loop 到自治团队

## 开场：为什么需要分层理解

很多人把 Claude Code 当成"带工具的 ChatGPT"，用了一段时间后发现不对劲：上下文越来越乱、工具越来越多但效果越来越差、规则越写越长却越不遵守。这不是 Prompt 问题，而是这套系统的设计就是这样的。

要真正用好 Claude Code，最直接的理解方式是把它拆成六层来看：**上下文层 → 控制层 → 工具层 → 验证层 → 协作层 → 持久化层**。只强化其中一层，系统就会失衡。CLAUDE.md 写太长，上下文先污染自己了；工具堆太多了，选择就搞不清楚了；subagents 开得到处都是，状态就漂移了。

这套架构不是凭空而来。2022 年的 ReAct 论文提出了"推理+行动"范式，奠定了 Agent Loop 的理论基础；2023 年的 Toolformer 和 Gorilla 论文则探索了大模型如何学会调用工具。Claude Code 的工程实践，正是这些学术思想落地的典范。

---

## 第一层：上下文层（200K 不是你想的那样）

很多人把上下文当"容量问题"，但卡住的地方通常不是不够长，而是太吵了——有用的信息被大量无关内容淹没了。

### 真实的上下文成本构成

Claude Code 的 200K 上下文并非全部可用：

```
200K 总上下文
├── 固定开销 (~15-20K)
│   ├── 系统指令: ~2K
│   ├── 所有启用的 Skill 描述符: ~1-5K
│   ├── MCP Server 工具定义: ~10-20K  ← 最大隐形杀手
│   └── LSP 状态: ~2-5K
│
├── 半固定 (~5-10K)
│   ├── CLAUDE.md: ~2-5K
│   └── Memory: ~1-2K
│
└── 动态可用 (~160-180K)
    ├── 对话历史
    ├── 文件内容
    └── 工具调用结果
```

一个典型 MCP Server（如 GitHub）包含 20-30 个工具定义，每个约 200 tokens，合计 4,000-6,000 tokens。接 5 个 Server，光这部分固定开销就到了 **25,000 tokens（12.5%）**。在要读大量代码的场景，这 12.5% 真的很关键。

### 推荐的上下文分层策略

```
始终常驻    → CLAUDE.md：项目契约 / 构建命令 / 禁止事项
按路径加载  → rules：语言 / 目录 / 文件类型特定规则
按需加载    → Skills：工作流 / 领域知识
隔离加载    → Subagents：大量探索 / 并行研究
不进上下文  → Hooks：确定性脚本 / 审计 / 阻断
```

偶尔用的东西就不要每次都加载进来，这是上下文治理的核心原则。

## 第二层：控制层（Hooks 和权限）

控制层的核心是 Hooks——在 Claude 执行操作前后，强制插入你自己的逻辑。Hooks 很容易被当成"自动运行的脚本"，但它更像是把一些不能交给 Claude 临场发挥的事情，重新收回到确定性的流程里。

### Hooks 的典型应用

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "pattern": "*.rs",
        "hooks": [{
          "type": "command",
          "command": "cargo check 2>&1 | head -30",
          "statusMessage": "Checking Rust..."
        }]
      }
    ]
  }
}
```

### 适合 vs 不适合放到 Hooks

| 适合 | 不适合 |
|------|--------|
| 阻断修改受保护文件 | 需要读大量上下文的复杂语义判断 |
| Edit 后自动格式化/lint | 长时间运行的业务流程 |
| SessionStart 后注入动态上下文 | 需要多步推理和权衡的决策 |

在 100 次编辑的会话中，每次节省 30-60 秒，累积节省 1-2 小时。注意限制输出长度（`| head -30`），避免 Hook 输出反而污染上下文。

## 第三层：工具层（dispatch map + 工具设计原则）

Claude Code 的核心不是"回答"，而是一个反复循环的代理过程：

```python
def agent_loop(query):
    messages = [{"role": "user", "content": query}]
    while True:
        response = client.messages.create(
            model=MODEL, system=SYSTEM, messages=messages,
            tools=TOOLS, max_tokens=8000,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return

        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        messages.append({"role": "user", "content": results})
```

不到 30 行，这就是整个智能体。后面 11 个章节都在这个循环上叠加机制——循环本身始终不变。

### dispatch map 设计

加工具不需要改循环，只需要把新工具注册进 dispatch map：

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

一个 lookup 替代任何 if/elif 链。循环永远不变，加工具 = 加 handler + 加 schema。

### 好工具 vs 坏工具

给 Claude 的工具和给人写的 API 不是一回事。给人用的 API 追求功能齐全，但给 agent 用，重点是让它更容易用对。

| 好工具设计 | 坏工具设计 |
|-----------|-----------|
| 名称前缀分层：`github_pr_*`、`jira_issue_*` | 一个工具覆盖多个领域 |
| 支持响应格式：`response_format: concise/detailed` | 总是返回完整输出 |
| 错误响应教模型如何修正 | 只抛 opaque error code |
| 合并成高层任务工具 | 暴露过多底层碎片工具 |

Claude Code 内部有个经典案例：AskUserQuestion 工具。第一版给已有工具加 question 参数，Claude 经常忽略；第二版用 markdown 格式约定，没有强制约束；第三版做成独立工具，调用即暂停，没有歧义，比前两版靠谱多了。

## 第四层：验证层（怎么知道做对了）

「Claude 说完成了」其实没啥用，你得能知道它做没做对、出了问题能退回来、过程还能查，这才算数。

### Verifier 的层级

- **最低层**：命令退出码、lint、typecheck、unit test
- **中间层**：集成测试、截图对比、contract test、smoke test
- **更高层**：生产日志验证、监控指标、人工审查清单

### 在 CLAUDE.md 中显式定义验证

```markdown
## Verification

For backend changes:
- Run `make test` and `make lint`
- For API changes, update contract tests under `tests/contracts/`

Definition of done:
- All tests pass
- Lint passes
- No TODO left behind unless explicitly tracked
```

一个简单判断：假如一个任务你都说不清楚「Claude 怎么才算做对了」，那它大概率也不适合直接丢给 Claude 自动完成。这正是 Reflexion 论文"自我反思修正"思想的工程实践。

## 第五层：协作层（Subagent + Teams）

### Subagent：派一个独立的 Claude 去干一件具体的事

Subagent 就是从主对话派出去的一个独立 Claude 实例，有自己的上下文窗口，只用你指定的工具，干完汇报结果。最大的价值不是"并行"，而是**隔离**。扫代码库、跑测试、做审查这类会产生大量输出的事，塞进主线程很快就把有效上下文挤没了，交给 Subagent 做，主线程只拿一个摘要，干净很多。

配置时要显式约束：
- `tools / disallowedTools`：限定能用什么工具
- `model`：探索任务用 Haiku/Sonnet，重要审查用 Opus
- `maxTurns`：防止跑飞

### Agent Teams：持久化队友 + JSONL 邮箱

真正的团队协作需要三样东西：能跨多轮对话存活的持久智能体、身份和生命周期管理、智能体之间的通信通道。

```
Communication:
  .team/
    config.json           <- team roster + statuses
    inbox/
      alice.jsonl         <- append-only, drain-on-read
      bob.jsonl
      lead.jsonl
```

MessageBus 采用 append-only 的 JSONL 收件箱：`send()` 追加一行；`read_inbox()` 读取全部并清空。每个队友在每次 LLM 调用前检查收件箱，将消息注入上下文。

## 第六层：持久化层（任务系统）

### 三层压缩机制

上下文窗口是有限的。读一个 1000 行的文件就吃掉 ~4000 token；读 30 个文件、跑 20 条命令，轻松突破 100k token。不压缩，智能体根本没法在大项目里干活。

三层压缩，激进程度递增：

```
[Layer 1: micro_compact]        (silent, every turn)
  Replace tool_result > 3 turns old with "[Previous: used {tool_name}]"
        |
        v
[Layer 2: auto_compact]
  Save transcript to .transcripts/
  LLM summarizes conversation. Replace all messages with [summary].
        |
        v
[Layer 3: compact tool]
  Model calls compact explicitly.
```

完整历史通过 transcript 保存在磁盘上。信息没有真正丢失，只是移出了活跃上下文。

### Prompt Caching：架构核心

Claude Code 的整个架构都是围绕 Prompt 缓存构建的。Prompt 缓存按前缀匹配工作，顺序很重要：

```
Claude Code 的 Prompt 顺序：
1. System Prompt → 静态，锁定
2. Tool Definitions → 静态，锁定
3. Chat History → 动态，在后面
4. 当前用户输入 → 最后
```

**会话中途不要切换模型**。Prompt 缓存是模型唯一的。已经和 Opus 对话了 100K tokens，切换到 Haiku 实际上比继续用 Opus 更贵，因为要为 Haiku 重建整个缓存。

**defer_loading 机制**：Claude Code 有数十个 MCP 工具，每次请求全量包含会很贵，但中途移除会破坏缓存。解决方案是发送轻量级 stub，标记 `defer_loading: true`。模型通过 ToolSearch 工具"发现"它们，完整的 schema 只在模型选择后才加载。

## 结尾：六层协同

用 Claude Code 大概会经历三个阶段：

1. **探索期**：当 ChatBot 用，问问题、跑命令
2. **掌握期**：发现它能改文件、开 PR、调 API
3. **治理期**：关注约束、验证、状态管理

到了第三阶段，关注点会悄悄变掉，从「这个功能怎么用」变成「怎么让 agent 在约束下自己跑起来」。

六层架构的核心思想：**上下文层管信息输入，控制层管权限边界，工具层管能力边界，验证层管质量闭环，协作层管任务分工，持久化层管状态延续**。每一层都有独立的治理原则，但真正强大的是六层协同。

最后分享两个实用的优化技巧：

1. **HANDOFF.md**：开新会话前，先让 Claude 写一份交接文档，把当前进度、尝试过什么、哪些走通了、哪些是死路写清楚。下一个 Claude 实例只读这个文件就能接着做，不依赖压缩算法的摘要质量。

2. **AI 审 AI**：进阶玩法是开一个 Claude 写计划，再开一个以"高级工程师"身份审这个计划。让 AI 审 AI，效果很好。

## 番外：扩展想法

展望未来，Agent 系统还有几个值得探索的方向：

- **真正的长期记忆**：不只是 200K，而是跨会话的记忆
- **跨会话学习和进化**：每次用都变得更懂你
- **多 Agent 知识共享**：队友之间共享学到的东西
- **自我调试和自我修复**：出问题自己排查自己修
- **主动式常驻助手**：从"踹一下动一下"到"自己找活干"

这些方向，正在从论文走向工程实践。
