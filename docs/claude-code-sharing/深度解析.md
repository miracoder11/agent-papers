## 一、技术实现深度

### 1.1 Prompt Caching 的具体机制

**前缀匹配原理**

Prompt 缓存是按前缀匹配工作的。Anthropic API 允许在消息的特定位置设置 `cache_control` 断点，从请求开头到每个断点之前的内容会被缓存：

```
Request 结构:
┌─────────────────────────────────────────────────────┐
│ System Prompt [cache_control]                       │ ← 缓存块 1
├─────────────────────────────────────────────────────┤
│ Tool Definitions [cache_control]                    │ ← 缓存块 2
├─────────────────────────────────────────────────────┤
│ Messages (对话历史)                                   │ ← 不缓存
│   - User: ...                                        │
│   - Assistant: ...                                   │
│   - User: ...                                        │
└─────────────────────────────────────────────────────┘
```

**关键约束**：
- 缓存必须从请求开头连续
- 中间不能有未缓存的断层
- 缓存是模型唯一的（Opus 和 Haiku 不能共享）

**Claude Code 的设计**：

```python
# Claude Code 的 Prompt 顺序设计
messages = [
    {"role": "system", "content": SYSTEM_PROMPT, "cache_control": {"type": "ephemeral"}},
    {"role": "system", "content": TOOL_DEFINITIONS, "cache_control": {"type": "ephemeral"}},
    # 动态部分不设置 cache_control
    *chat_history,
    {"role": "user", "content": current_input}
]
```

**为什么这个顺序很重要？**

如果把动态内容（如当前时间）放在 System Prompt 里，每次请求都变，缓存就废了。Claude Code 的做法是把动态信息放到 `<system-reminder>` 标签里，作为 user 消息的一部分传入，System Prompt 保持静态。

### 1.2 defer_loading：工具的延迟加载

**问题**：Claude Code 有数十个 MCP 工具，每次请求全量包含会很贵。但中途增删工具会破坏缓存。

**解决方案**：

```python
# 发送轻量级 stub
tools = [
    {
        "name": "mcp_github_create_issue",
        "description": "Create a GitHub issue",
        "defer_loading": True  # 关键标记
    },
    # ... 其他 stub
]

# 完整的 schema 只有在模型选择后才加载
def load_full_tool_schema(tool_name):
    return FULL_TOOL_SCHEMAS[tool_name]
```

**工作流程**：

```
1. 初始请求只包含 stub（工具名 + 简短描述）
2. 模型通过 ToolSearch 工具"发现"完整工具列表
3. 模型选择某个工具后，系统加载完整 schema
4. 执行工具调用

好处：
- 缓存前缀保持稳定（stub 不变）
- 减少初始请求的 token 消耗
- 按需加载，不影响功能
```

### 1.3 上下文压缩的实现细节

**三层压缩的具体代码**：

```python
# Layer 1: micro_compact - 每次调用前执行
def micro_compact(messages: list, keep_recent: int = 3) -> list:
    """将旧的 tool_result 替换为占位符"""
    tool_results = []
    for i, msg in enumerate(messages):
        if msg["role"] == "user" and isinstance(msg.get("content"), list):
            for j, part in enumerate(msg["content"]):
                if isinstance(part, dict) and part.get("type") == "tool_result":
                    tool_results.append((i, j, part))

    if len(tool_results) <= keep_recent:
        return messages

    # 替换旧结果
    for _, _, part in tool_results[:-keep_recent]:
        if len(part.get("content", "")) > 100:
            part["content"] = f"[Previous: used {part.get('tool_name', 'tool')}]"

    return messages


# Layer 2: auto_compact - token 超过阈值时触发
def auto_compact(messages: list, threshold: int = 50000) -> list:
    """保存完整对话，让 LLM 做摘要"""
    # 1. 保存完整历史到磁盘
    transcript_path = TRANSCRIPT_DIR / f"transcript_{int(time.time())}.jsonl"
    with open(transcript_path, "w") as f:
        for msg in messages:
            f.write(json.dumps(msg, default=str) + "\n")

    # 2. 调用 LLM 做摘要（这一步命中缓存，价格是 1/10）
    response = client.messages.create(
        model=MODEL,
        messages=[{
            "role": "user",
            "content": "Summarize this conversation for continuity. "
                       "Preserve: architecture decisions, key changes, "
                       "verification status, open TODOs.\n\n"
                       + json.dumps(messages, default=str)[:80000]
        }],
        max_tokens=4000,
    )

    # 3. 替换消息历史
    return [
        {"role": "user", "content": f"[Compressed]\n\n{response.content[0].text}"},
        {"role": "assistant", "content": "Understood. Continuing."},
    ]
```

**陷阱**：默认压缩算法按"可重新读取"判断，会优先删除：
- 早期的 Tool Output
- 文件内容
- **顺带把架构决策和约束理由也扔了**

**解决方案**：在 CLAUDE.md 里写 Compact Instructions：

```markdown
## Compact Instructions

When compressing, preserve in priority order:

1. Architecture decisions (NEVER summarize)
2. Modified files and their key changes
3. Current verification status (pass/fail)
4. Open TODOs and rollback notes
5. Tool outputs (can delete, keep pass/fail only)
```

### 1.4 dispatch map 的设计

**为什么用 dispatch map 而不是 if/elif？**

```python
# 反模式：if/elif 链
def execute_tool(name, input):
    if name == "bash":
        return run_bash(input["command"])
    elif name == "read_file":
        return run_read(input["path"])
    elif name == "write_file":
        return run_write(input["path"], input["content"])
    # ... 每加一个工具都要改这里

# 正确模式：dispatch map
TOOL_HANDLERS = {
    "bash": lambda **kw: run_bash(kw["command"]),
    "read_file": lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file": lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}

def execute_tool(name, input):
    handler = TOOL_HANDLERS.get(name)
    if handler:
        return handler(**input)
    return f"Unknown tool: {name}"
```

**好处**：
- 加工具 = 加 handler + 加 schema，循环不用动
- 一个 lookup 替代任何 if/elif 链
- 可以动态增删工具

### 1.5 身份重注入

**问题**：上下文压缩后，智能体可能忘了自己是谁。

```python
# s11 的解决方案
def check_and_reinject_identity(messages: list, name: str, role: str, team: str):
    """如果消息过短，说明发生了压缩，需要重注入身份"""
    if len(messages) <= 3:  # 压缩后通常只有 2-3 条
        messages.insert(0, {
            "role": "user",
            "content": f"<identity>You are '{name}', role: {role}, team: {team}. "
                       f"Continue your work.</identity>"
        })
        messages.insert(1, {
            "role": "assistant",
            "content": f"I am {name}. Continuing."
        })
    return messages
```

**为什么这个设计有效？**

压缩后的消息通常是：
```
[Compressed]
...摘要内容...

Understood. Continuing.
```

只有 2 条消息。如果检测到这种情况，就在开头插入身份块，让智能体"想起"自己是谁。

## 二、设计哲学深度

### 2.1 AskUserQuestion 工具的三版迭代

**问题**：需要让 Claude 在任务中途停下来问用户。

**第一版：加参数**

```python
# 给 Bash 工具加一个 question 参数
def run_bash(command: str, question: str = None):
    if question:
        # 期望 Claude 会停下来问
        pass
    # 执行命令...
```

**结果**：Claude 大多数时候直接忽略 question 参数，继续往下跑。

**为什么？** 因为这不是一个"强制"机制，只是一个"建议"。模型没有理由一定要遵守。

**第二版：约定输出格式**

```python
# 要求 Claude 按特定格式输出
# 外层解析到这个格式就暂停

# Claude 被期望输出：
# <question>Should I proceed?</question>
```

**结果**：Claude 经常"忘了"按格式写，提问逻辑非常脆弱。

**为什么？** 同样的问题——没有强制约束，模型输出是自由的。

**第三版：独立工具**

```python
# 做成独立的 AskUserQuestion 工具
tools = [
    {
        "name": "AskUserQuestion",
        "description": "Ask the user a question and wait for response",
        "input_schema": {
            "type": "object",
            "properties": {
                "question": {"type": "string"},
                "options": {"type": "array", "items": {"type": "string"}}
            },
            "required": ["question"]
        }
    }
]
```

**结果**：Claude 想提问就必须显式调用它，调用即暂停，没有歧义。

**设计教训**：

```
想让 Claude 做什么，就给它一个专门的工具。

加参数、约定格式，都是"软约束"，模型可以忽略。
独立工具是"硬约束"，调用即触发，无法绕过。
```

### 2.2 Todo 工具的教训：从限制变枷锁

**背景**：早期模型不够强，容易忘记任务。Claude Code 团队加了 TodoWrite 工具 + 每 5 轮插入提醒。

```python
# 每 5 轮插入提醒
if turn_count % 5 == 0:
    messages.append({
        "role": "user",
        "content": "Reminder: Your current todo list is:\n" + todo_list
    })
```

**结果**：随着模型变强，这个工具反而成了限制。

**为什么？**
- Todo 提醒让 Claude 认为必须严格遵循
- 无法灵活修改计划
- 模型变强后，它反而变成了枷锁

**教训**：

```
当初加这个工具是因为模型不够强。
模型变强之后，它反而变成了枷锁。

值得过段时间回来检查：当初加的限制还成不成立？
```

### 2.3 搜索工具的演进：RAG → Grep → 渐进式披露

**第一版：RAG 向量数据库**

```python
# 用向量数据库做语义搜索
def search_code(query: str):
    embeddings = get_embeddings(query)
    results = vector_db.search(embeddings, top_k=10)
    return results
```

**问题**：
- 需要预先索引
- 不同环境脆弱
- 最重要的是：**Claude 不喜欢用**

**第二版：Grep 工具**

```python
# 让 Claude 自己搜索
def grep(pattern: str, path: str):
    return subprocess.run(["grep", "-r", pattern, path])
```

**结果**：好用很多。

**为什么 Claude 更喜欢 Grep？**

因为 Grep 是它熟悉的概念，可以直接表达"我要找什么"。RAG 需要先理解"语义搜索"是什么，然后猜测返回结果的相关性。

**意外的收获：渐进式披露**

```python
# Claude 读 Skill 文件
skill_content = read_file("skills/code-review/SKILL.md")

# SKILL.md 里引用了其他文件
# "See examples in examples.md"
# Claude 会递归读取

examples = read_file("skills/code-review/examples.md")
```

模型会递归读取，按需发现信息，不需要提前塞进去。这个模式后来被叫做"渐进式披露"（Progressive Disclosure）。

### 2.4 Subagent：为什么强调隔离而不是并行？

**直觉**：Subagent 是用来并行的，对吧？

**实际情况**：Subagent 的最大价值是隔离，不是并行。

**为什么？**

```
扫代码库、跑测试、做审查这类事：
- 会产生大量输出（几百 KB）
- 输出本身价值不高（"这个项目用 pytest"）
- 但会挤掉上下文里的有效信息

如果直接在主对话里做：
- messages 数组越来越胖
- 有效上下文被噪声淹没
- 后续决策质量下降

用 Subagent：
- 子智能体跑 30+ 次工具调用
- 整个消息历史直接丢弃
- 主对话只拿一个摘要
```

**并行是附赠品**，隔离才是核心价值。

### 2.5 工具设计原则：给人用的 API vs 给 Agent 用的工具

**给人用的 API**：
- 追求功能齐全
- 详细的文档
- 灵活的参数组合

**给 Agent 用的工具**：
- 追求"容易用对"
- 清晰的错误提示（教它怎么修正）
- 有限的选项（减少决策复杂度）

**例子**：

```python
# 反模式：给 Agent 太多选择
tools = [
    {"name": "list_all_repos", ...},
    {"name": "list_all_issues", ...},
    {"name": "list_all_prs", ...},
    {"name": "list_all_users", ...},
]
# Agent 要自己决定用哪个，容易选错

# 正确模式：合并成高层任务工具
tools = [
    {"name": "github_find_stale_prs",
     "description": "Find PRs with no activity for 30+ days",
     ...},
]
```

**设计原则**：
1. 名称前缀按系统分层：`github_pr_*`、`jira_issue_*`
2. 对大响应支持 `response_format: concise / detailed`
3. 错误响应要教模型如何修正，不要只抛 opaque error code
4. 能合并成高层工具时，不要暴露过多底层碎片工具

## 三、学术理论深度

### 3.1 ReAct 论文与 Agent Loop

**论文**：ReAct: Synergizing Reasoning and Acting in Language Models (2022)

**核心思想**：让 LLM 在"推理"和"行动"之间交替，而不是一次性输出答案。

```
ReAct 循环：
Thought: 我需要先了解项目结构
Action: list_files
Observation: [文件列表]
Thought: 看到有 tests/ 目录，应该有测试
Action: read_file tests/test_main.py
Observation: [测试文件内容]
Thought: 这是一个 pytest 项目...
```

**与 Claude Code 的对应**：

| ReAct 概念 | Claude Code 实现 |
|-----------|-----------------|
| Thought | Claude 的内部推理 |
| Action | Tool 调用 |
| Observation | Tool Result |
| 循环 | while stop_reason == "tool_use" |

**30 行 Agent Loop 就是 ReAct 的最小实现**：

```python
def agent_loop(query):
    messages = [{"role": "user", "content": query}]
    while True:
        # Thought + Action
        response = client.messages.create(
            model=MODEL, system=SYSTEM,
            messages=messages, tools=TOOLS,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return  # 循环结束

        # Observation
        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_tool(block.name, block.input)
                results.append({"type": "tool_result", "content": output})
        messages.append({"role": "user", "content": results})
```

### 3.2 Toolformer 与工具学习

**论文**：Toolformer: Language Models Can Teach Themselves to Use Tools (2023)

**核心思想**：让模型学会"什么时候该用工具"、"怎么用工具"。

**关键发现**：
- 模型可以学会在合适的位置插入 API 调用
- 不需要大量人工标注
- 用"自监督"方式学习

**与 Claude Code 的对应**：

Claude Code 的工具使用能力来自两方面：
1. **预训练**：模型见过大量 API 调用代码
2. **Fine-tuning**：Anthropic 对模型做了工具使用的对齐

**dispatch map 的设计让工具学习变得简单**：

```python
# 工具的 schema 定义了"怎么用"
tools = [
    {
        "name": "read_file",
        "description": "Read a file from the filesystem",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string", "description": "File path"},
                "limit": {"type": "integer", "description": "Max lines"}
            },
            "required": ["path"]
        }
    }
]
```

模型通过 schema 理解工具的语义，然后决定何时调用、传什么参数。

### 3.3 Reflexion 与自我反思

**论文**：Reflexion: Language Agents with Verbal Reinforcement Learning (2023)

**核心思想**：让模型在失败后"反思"，然后重试。

```
Reflexion 循环：
1. 尝试任务
2. 如果失败，生成"反思"：为什么失败？应该怎么做？
3. 把反思加入上下文，重试
```

**与 Claude Code 的对应**：

Claude Code 的"验证闭环"就是 Reflexion 的工程实现：

```markdown
## Verification

For backend changes:
- Run `make test` and `make lint`
- If tests fail:
  1. Read the error message
  2. Understand what went wrong
  3. Fix the issue
  4. Re-run tests

Definition of done:
- All tests pass
- Lint passes
```

**关键设计**：
- 验证标准要提前定义（不能等出问题再想）
- 失败后要有明确的修复路径
- 要能"撤回"到上一个状态

### 3.4 多智能体协作的理论基础

**相关论文**：
- MetaGPT (2023): 多角色协作框架
- AutoGen (2023): 自动化多智能体对话
- CAMEL (2023): 角色扮演的协作

**核心思想**：把复杂任务分解给多个"专家"角色，通过协作完成。

**Claude Code 的实现**：

```
s09 Agent Teams:
- 领导：分配任务、协调进度
- 程序员：写代码
- 测试员：跑测试

s11 自治智能体:
- 队友自己扫描任务看板
- 认领没人做的任务
- 做完再找下一个
```

**JSONL 邮箱的设计**：

```python
class MessageBus:
    def send(self, sender, to, content):
        msg = {"from": sender, "content": content, "timestamp": time.time()}
        with open(f"{to}.jsonl", "a") as f:
            f.write(json.dumps(msg) + "\n")  # append-only

    def read_inbox(self, name):
        path = f"{name}.jsonl"
        msgs = [json.loads(l) for l in open(path)]
        open(path, "w").close()  # drain - 读后清空
        return msgs
```

**为什么用 JSONL？**
- append-only：保证消息不丢
- drain-on-read：读后清空，避免重复处理
- 简单可靠：不需要复杂的消息队列

## 四、实践经验深度

### 4.1 真实踩过的坑

**坑 1：CLAUDE.md 写太长**

```
问题：写了 500 行 CLAUDE.md，Claude 反而不遵守了

原因：
- CLAUDE.md 占用 10K tokens
- 每次请求都要加载
- 关键信息被稀释了

解决：
- 砍到 50 行（~2K tokens）
- 只保留：build/test 命令、架构边界、禁止事项
- 其他内容移到 Skills 或 rules
```

**坑 2：MCP Server 接太多**

```
问题：接了 8 个 MCP Server，上下文经常爆

原因：
- 每个 Server ~5000 tokens
- 8 个就是 40000 tokens（20%）
- 还没开始干活就用了 1/5

解决：
- 只接常用的 3 个
- 用 `/mcp` 断开闲置的
- 需要时再连
```

**坑 3：Subagent 权限太宽**

```
问题：Subagent 和主线程一样权限，隔离没有意义

原因：
- 没有配置 disallowedTools
- Subagent 可以修改任何文件

解决：
- 明确限定 Subagent 能用的工具
- 探索任务只给 Read/Grep
- 需要写文件时用 worktree 隔离
```

**坑 4：压缩后忘了架构决策**

```
问题：两小时后改代码，Claude 不记得两小时前定了什么

原因：
- 压缩算法按"可重新读取"判断
- 早期的架构决策被删了

解决：
- 在 CLAUDE.md 里写 Compact Instructions
- 明确"架构决策永不摘要"
- 或者写 HANDOFF.md 主动交接
```

### 4.2 边界条件

**什么时候不该用 Subagent？**

- 子任务之间强依赖，频繁要共享中间状态
- 输出格式不固定，主线程拿到没法用
- 任务本身很简单，开 Subagent 反而增加开销

**什么时候不该加 Tool？**

- 本地 shell 可以可靠完成的事情
- 模型只需要静态知识，不需要真正与外部交互
- 需求更适合 Skill 的工作流约束

**什么时候用 /clear vs /compact？**

- 任务切换：用 `/clear`（完全清空）
- 同一任务进入新阶段：用 `/compact`（保留摘要）

### 4.3 意外发现

**发现 1：Claude 不喜欢用 RAG**

改成 Grep 工具后，好用很多。Claude 更喜欢直接表达"我要找什么"，而不是理解"语义搜索"的概念。

**发现 2：AI 审 AI 效果很好**

开一个 Claude 写计划，再开一个以"高级工程师"身份审这个计划，能发现很多问题。

**发现 3：环境透明比想象中重要**

Claude Code 调用的都是真实的 shell、git、package manager。只要有一层不透明，它就只能开始猜，一猜可靠性就掉。

解决方案：加 `doctor` 命令，先收集环境状态，输出结构化报告。

### 4.4 实战经验补充

#### 经验 1：Context 管理的隐形杀手

**问题**：MCP Server 占了很大篇幅，导致频繁 auto compact。

```
场景：
- 接了 5 个 MCP Server
- 每个 Server ~5000 tokens
- 还没开始干活，25% 上下文就没了
- 稍微读几个文件就触发 auto compact

后果：
- 对话质量急剧下降
- Claude 开始"忘记"之前讨论的内容
- 需要反复重新解释需求
```

**核心认知**：不理解 Context 的构成，就不知道怎么管理它。

```
200K 上下文的真实构成：

固定开销（不可控）：
├── 系统指令: ~2K
├── MCP Server 工具定义: ~10-20K  ← 最大隐形杀手
└── LSP 状态: ~2-5K

半固定（可控）：
├── CLAUDE.md: ~2-5K
└── Memory: ~1-2K

动态可用：
├── 对话历史
├── 文件内容
└── 工具调用结果
```

**实践建议**：
1. 用 `/context` 定期检查上下文占用
2. 只接必要的 MCP Server
3. 长会话主动用 `/compact`，别等系统自动触发

#### 经验 2：Context 丢失的严重后果

**问题**：技术架构、Plan 记在 Context 里，一轮 compact 后全忘了。

```
典型场景：
1. 和 Claude 讨论了半小时架构设计
2. 确定了技术方案、模块划分、接口定义
3. 继续写代码，Context 越来越长
4. 触发 auto compact
5. 继续改代码时，Claude 已经不记得架构决策了
6. 开始写和之前决策矛盾的代码
```

**根本原因**：Context 不是持久化存储，它是临时的工作记忆。

**重要认知**：

```
任何重要的东西，都不要指望 Claude 能记住。

Plan、技术架构、设计决策：
- 必须记录到文档里
- 让 Claude 读文档，而不是记忆

Context 的正确用法：
- 放当前任务的上下文
- 放最近的操作历史
- 不放长期的知识和决策
```

**解决方案**：

1. **讨论完架构，立刻写文档**

```markdown
# Architecture Decision Record

## 2024-03-17: 模块划分

- 用户模块：src/user/
- 订单模块：src/order/
- 支付模块：src/payment/

接口定义：
- UserService.createUser()
- OrderService.createOrder()
- PaymentService.processPayment()
```

2. **下一个任务开始前，让 Claude 读文档**

```
"继续开发用户模块。先读 docs/architecture.md 了解架构设计。"
```

3. **用 HANDOFF.md 做会话交接**

```markdown
# HANDOFF

## 已完成
- 用户注册功能
- 用户登录功能

## 进行中
- 用户信息修改：已写了一半，在 src/user/service.py

## 技术决策
- 密码用 bcrypt 加密
- Session 存 Redis，过期时间 7 天

## 下一步
- 完成用户信息修改
- 写用户信息修改的测试
```

#### 经验 3：验证机制是工程化的关键

**核心认知**：工作中最重要的两件事：

1. **提供足够的上下文**
2. **做好验证机制**

**验证的重要性**：

```
没有验证：
- Claude 说"完成了"
- 你不知道它做没做对
- 出问题不知道哪里挂的
- 没法撤回

有验证：
- 明确知道"什么叫做完"
- 出问题立刻发现
- 有回退路径
```

**实践做法**：

```markdown
## Verification

端到端测试：
- 从用户使用角度测试
- 不只测单元，测整个流程

回归测试：
- 改了 A 功能，确保 B 功能没坏
- 自动化运行所有测试

验证回路：
1. Claude 完成任务
2. 自动跑测试
3. 测试失败 → Claude 读错误 → 修复 → 重跑
4. 测试通过 → 确认完成
```

**我的实践**：

花了很多时间保证端到端测试可用，并做好回归测试。这样 Claude 每次改动后，能立刻知道有没有破坏现有功能。

#### 经验 4：Skill 的正确使用时机

**核心认知**：Skill 的最大好处是把重复的东西固定下来、模板化。

**Skill 的优势**：
- 不是代码，需求变动时不需要改
- 有很强的泛用性
- 可以跨项目复用

**使用建议**：

```
✅ 有重复操作时，才考虑使用 Skill

❌ 不要为了使用 Skill 而使用 Skill

正确流程：
1. 先手动做几遍，确认流程稳定
2. 发现确实在重复
3. 再封装成 Skill

错误流程：
1. 看到别人用 Skill
2. 自己也想用
3. 强行封装一个不确定是否有用的 Skill
```

**判断标准**：

| 场景 | 是否适合做 Skill |
|------|-----------------|
| 每次发布都要跑的检查清单 | ✅ 适合 |
| 偶尔用的调试流程 | ❌ 不适合，写成文档 |
| 固定的代码审查步骤 | ✅ 适合 |
| 只在某个项目用的配置 | ❌ 不适合，放 CLAUDE.md |

**总结**：有问题再去用 Skill，不是为了用而用。

## 五、扩展与未来

### 5.1 当前局限

1. **上下文还是有限的**：200K 不够用，需要更好的长期记忆
2. **没有跨会话学习**：每次会话都是"新"的 Claude
3. **协作是单向的**：Subagent 只能向主对话汇报，不能主动提问

### 5.2 可能的优化方向

**长期记忆**：
- 把重要决策存到文件（HANDOFF.md）
- 下次会话加载
- 不依赖模型的记忆

**跨会话学习**：
- 记录"什么有效、什么无效"
- 形成个人知识库
- 每个 Claude 实例都能访问

**主动式助手**：
- 参考 OpenClaw 的设计
- 心跳机制：每 30 秒醒一次，检查有没有事
- 定时任务：到点自动执行

### 5.3 开源工具推荐

- **RTK (Rust Token Killer)**: 减少 Tool Output 噪声
- **claude-health**: 配置健康检查
- **learn-claude-code**: 从零构建 Agent 的教程
- **Kode-cli**: 开源的 Claude Code 替代品
