# Claude Code 实践指南：从会用到用好

## 开场：从会用到用好

很多人刚开始用 Claude Code 时，会把它当成一个更聪明的 ChatBot。但很快就会发现不对劲：上下文越来越乱、工具越来越多但效果越来越差、规则越写越长却越不遵守。

这不是 Prompt 的问题，而是这套系统的设计就是这样。用了一段时间才意识到，卡住的地方几乎从来不是模型不够聪明，更多时候是给了它错误的上下文，或者写出来了但根本没法判断对不对。

今天这篇文章不讲概念，只讲实践——怎么用好 Claude Code，避开那些常见的坑。

---

## 第一部分：理解底层——30 行代码看清本质

Claude Code 的核心不是"回答"，而是一个反复循环的代理过程。理解这一点，很多问题就好排查了。

```python
def agent_loop(query):
    messages = [{"role": "user", "content": query}]
    while True:
        response = client.messages.create(
            model=MODEL, system=SYSTEM, messages=messages,
            tools=TOOLS, max_tokens=8000,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return

        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        messages.append({"role": "user", "content": results})
```

不到 30 行，这就是整个智能体。一个循环、一个退出条件、一个工具调用——后面所有的机制都在这个循环上叠加。

加工具呢？只需要一个 dispatch map：

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

循环不用动，新工具注册进 dispatch map 就行。理解了这个设计，你就明白为什么有时候 Claude 选错工具——不是它笨，而是工具描述没写清楚。

## 第二部分：上下文工程——最重要的系统约束

很多人把上下文当"容量问题"，但卡住的地方通常不是不够长，而是太吵了——有用的信息被大量无关内容淹没了。

### 真实的上下文成本构成

Claude Code 的 200K 上下文并非全部可用：

```text
200K 总上下文
├── 固定开销 (~15-20K)
│   ├── 系统指令: ~2K
│   ├── 所有启用的 Skill 描述符: ~1-5K
│   ├── MCP Server 工具定义: ~10-20K  ← 最大隐形杀手
│   └── LSP 状态: ~2-5K
│
├── 半固定 (~5-10K)
│   ├── CLAUDE.md: ~2-5K
│   └── Memory: ~1-2K
│
└── 动态可用 (~160-180K)
    ├── 对话历史
    ├── 文件内容
    └── 工具调用结果
```

一个典型 MCP Server（如 GitHub）包含 20-30 个工具定义，每个约 200 tokens，合计 4,000-6,000 tokens。接 5 个 Server，光这部分固定开销就到了 25,000 tokens（12.5%）。在要读大量代码的场景，这 12.5% 真的很关键。

### 推荐的上下文分层策略

```text
始终常驻    → CLAUDE.md：项目契约 / 构建命令 / 禁止事项
按路径加载  → rules：语言 / 目录 / 文件类型特定规则
按需加载    → Skills：工作流 / 领域知识
隔离加载    → Subagents：大量探索 / 并行研究
不进上下文  → Hooks：确定性脚本 / 审计 / 阻断
```

说白了，偶尔用的东西就不要每次都加载进来。

### 减少噪声的实用技巧

**RTK 思路**：Tool Output 是另一个隐形杀手。`cargo test` 一次完整输出动辄几千行，Claude 并不需要全看。只留决策需要的核心信息：

```text
# Claude 看到的原始输出
running 262 tests
test auth::test_login ... ok
...（几千行）

# 走 RTK 之后
✓ cargo test: 262 passed (1 suite, 0.08s)
```

**手动截断**：每条命令加 `| head -30`，避免输出污染上下文。

## 第三部分：工具和能力——Skills 与 Subagents 的正确用法

### Skills 三种典型类型

**类型一：检查清单型（质量门禁）**

发布前跑一遍，确保不漏项：

```yaml
---
name: release-check
description: Use before cutting a release to verify build, version, and smoke test.
---

## Pre-flight (All must pass)
- [ ] `cargo build --release` passes
- [ ] `cargo clippy -- -D warnings` clean
- [ ] Version bumped in Cargo.toml
- [ ] CHANGELOG updated
```

**类型二：工作流型（标准化操作）**

配置迁移高风险，显式调用 + 内置回滚步骤：

```yaml
---
name: config-migration
description: Migrate config schema. Run only when explicitly requested.
disable-model-invocation: true
---

## Steps
1. Backup: `cp ~/.config/kaku/config.toml ~/.config/kaku/config.toml.bak`
2. Dry run: `kaku config migrate --dry-run`
3. Apply: remove `--dry-run` after confirming output
```

**类型三：领域专家型（封装决策框架）**

运行时出问题时让 Claude 按固定路径收集证据：

```yaml
---
name: runtime-diagnosis
description: Use when kaku crashes, hangs, or behaves unexpectedly at runtime.
---

## Evidence Collection
1. Run `kaku doctor` and capture full output
2. Last 50 lines of `~/.local/share/kaku/logs/`

## Decision Matrix
| Symptom | First Check |
|---|---|
| Crash on startup | doctor output → Lua syntax error |
| Rendering glitch | GPU backend / terminal capability |
```

### Subagent 配置建议

Subagent 最大的价值不是"并行"，而是隔离。扫代码库、跑测试这类会产生大量输出的事，交给 Subagent 做，主线程只拿一个摘要。

配置时要显式约束：

- **tools / disallowedTools**：限定能用什么工具，别给和主线程一样宽的权限
- **model**：探索任务用 Haiku/Sonnet，重要审查用 Opus
- **maxTurns**：防止跑飞
- **isolation: worktree**：需要动文件时隔离文件系统

### Subagent 常见反模式

- 子代理权限和主线程一样宽，隔离没有意义
- 输出格式不固定，主线程拿到没法用
- 子任务之间强依赖，频繁要共享中间状态

## 第四部分：控制和治理——Hooks 与反模式

### Hooks 适合 vs 不适合

**适合**：
- 阻断修改受保护文件
- Edit 后自动格式化/lint/轻量校验
- SessionStart 后注入动态上下文（Git 分支、环境变量）
- 任务完成后推送通知

**不适合**：
- 需要读大量上下文的复杂语义判断
- 长时间运行的业务流程
- 需要多步推理和权衡的决策

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "pattern": "*.rs",
        "hooks": [{
          "type": "command",
          "command": "cargo check 2>&1 | head -30",
          "statusMessage": "Checking Rust..."
        }]
      }
    ]
  }
}
```

注意限制输出长度（`| head -30`），避免 Hook 输出反而污染上下文。

### 常见反模式一览

| 反模式 | 后果 | 解决方案 |
|--------|------|----------|
| CLAUDE.md 写太长 | 自己污染自己，先占上下文 | 保持短、硬、可执行 |
| 工具堆太多 | 选择困难，容易选错 | 按场景收敛起名 |
| Subagent 权限太宽 | 隔离无意义 | 显式限制 tools |
| 有副作用的 Skill 允许自动调用 | 不可控行为 | 设置 disable-model-invocation |
| Skill 描述过短 | 触发条件模糊 | 写清楚"何时该用我" |
| 会话中途切换模型 | 破坏缓存，反而更贵 | 用 Subagent 交接 |

## 第五部分：日常命令——高频命令的使用场景

这些命令说白了就干一件事：主动管理上下文，别等系统自己处理。

### 上下文管理

```bash
/context   # 查看 token 占用结构，排查 MCP 和文件读取占比
/clear     # 清空会话，同一问题被纠偏两次以上就重来
/compact   # 压缩但保留重点，配合 Compact Instructions
/memory    # 确认哪些 CLAUDE.md 真的被加载了
```

### 能力与治理

```bash
/mcp        # 管理 MCP 连接，检查 token 成本，断开闲置 server
/hooks      # 管理 hooks，控制平面入口
/permissions # 查看或更新权限白名单
/model      # 切换模型：Opus 用于深度推理，Sonnet 用于常规，Haiku 用于快速探索
```

### 会话连续性

```bash
claude --continue            # 恢复当前目录最近会话
claude --resume              # 打开选择器恢复历史会话
claude --continue --fork     # 从已有会话分叉，同一起点不同方案
claude -p "prompt"           # 非交互模式，接入 CI / pre-commit
```

### 进阶技巧

- **AI 审 AI**：开一个 Claude 写计划，再开一个以"高级工程师"身份审这个计划
- **Plan Mode**：按两下 Shift+Tab，把探索和执行拆开
- **HANDOFF.md**：开新会话前，让 Claude 写一份交接文档，包含进度、尝试过什么、哪些走通了
- **/health**：一键检查配置状态，输出优先级报告

## 结尾：行动清单

听完这篇分享，你可以立刻做这几件事：

1. **跑一次 /health**：检查你的配置现在处于什么状态
2. **精简 CLAUDE.md**：删掉空泛原则和大段背景，只保留命令、约束、架构边界
3. **检查 MCP 工具**：看看是否真的需要那么多 MCP Server，每个占多少 token
4. **配置一个 Hook**：比如 Edit 后自动跑 lint，加 `| head -30` 截断
5. **写一个 Skill**：从检查清单型开始，发布前跑一遍

## 附录：常见问题 Q&A

**Q1：CLAUDE.md 应该写多长？**

Anthropic 官方自己的 CLAUDE.md 大约只有 2.5K tokens。保持短、硬、可执行。优先写命令、约束、架构边界。大段背景介绍、完整 API 文档、空泛原则都不该放进去。

**Q2：什么时候该用 Subagent？**

当任务会产生大量输出（扫代码库、跑测试、做审查），但这些输出主线程只需要一个摘要时。Subagent 的价值是隔离，不是并行。

**Q3：压缩机制有什么坑？**

默认压缩算法按"可重新读取"判断，早期的 Tool Output 和文件内容会被优先删掉，顺带把架构决策和约束理由也一起扔了。解决方案是在 CLAUDE.md 里写明 Compact Instructions：

```markdown
## Compact Instructions
When compressing, preserve in priority order:
1. Architecture decisions (NEVER summarize)
2. Modified files and their key changes
3. Current verification status (pass/fail)
4. Open TODOs and rollback notes
```

**Q4：Plan Mode 什么时候用？**

复杂重构、迁移、跨模块改动。按两下 Shift+Tab 进入，把探索和执行拆开。进阶玩法是开一个 Claude 写计划，再开一个审这个计划。

**Q5：会话中途能切换模型吗？**

不推荐。Prompt 缓存是模型唯一的，切换模型要重建整个缓存，实际上比继续用原模型更贵。确实需要切换的话，用 Subagent 交接。
