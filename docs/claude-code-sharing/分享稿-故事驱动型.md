# 当我发现 Claude Code 的设计哲学

## 开场：翻车现场

说实话，刚开始用 Claude Code 的时候，我挺崩溃的。

一开始特别顺，越用越觉得它"懂我"。但过了半小时，它开始"犯蠢"——明明刚才说过的事，它忘了；明明知道的项目约定，它不遵守。

然后我开始写更长的 CLAUDE.md，加更多的规则，接更多的 MCP 工具……结果呢？**上下文越来越乱、工具越来越多但效果越来越差、规则越写越长却越不遵守。**

折腾了一段时间之后我才意识到：这不是 Prompt 的问题，而是这套系统的设计就是这样的。

今天我想和大家分享的，是我这半年深度使用 Claude Code 之后，逐渐理解到的设计哲学——每一个看似"不合理"的设计背后，都有一个真实的故事。

---

## 第一幕：原来如此

### 从一个简单的循环开始

理解 Claude Code 的起点，其实只需要看一段代码。

```python
def agent_loop(query):
    messages = [{"role": "user", "content": query}]
    while True:
        response = client.messages.create(
            model=MODEL, system=SYSTEM,
            messages=messages, tools=TOOLS,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return

        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        messages.append({"role": "user", "content": results})
```

**不到 30 行，这就是整个智能体。**

一个循环 + 一个退出条件 + 一个工具执行器。2022 年的 ReAct 论文把这个叫做"推理+行动"范式，后来 Toolformer 论文又证明了语言模型可以学会使用工具。但落到代码层面，就是这么简单的东西。

后面所有复杂的机制——Subagent、Skills、Hooks、Agent Teams——都是在这个循环上叠加的，**循环本身始终不变**。

我第一次看到这个的时候，有一种"原来如此"的感觉。原来那些看起来很神奇的 AI 编程助手，核心就是：让模型反复地"想——做——看结果——再想"。

### dispatch map：工具的优雅

第二个让我恍然大悟的，是工具的设计方式。

我之前以为加工具要改很多地方。但 Claude Code 的做法非常优雅：

```python
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

一个字典，工具名映射到处理函数。循环里按名字查找就行。加工具 = 加 handler + 加 schema，循环永远不动。

这让我想起那句格言：**"加一个工具，只加一个 handler"**。

## 第二幕：设计的故事

### AskUserQuestion：三版迭代

真正让我感受到"设计是有原因的"，是 AskUserQuestion 工具的故事。

Claude Code 团队遇到了一个问题：智能体在任务中途需要停下来问用户。比如"你想用哪个框架？"或者"这个错误怎么处理？"

他们前后试了三种做法：

**第一版**：给已有工具（如 Bash）加一个 question 参数，让 Claude 在调用工具时顺带提问。

结果：Claude 大多数时候直接忽略这个参数，继续往下跑，根本不停下来问。

**第二版**：要求 Claude 在输出里写特定 markdown 格式，外层解析到这个格式就暂停。

问题：没有强制约束，Claude 经常"忘了"按格式写，提问逻辑非常脆弱。

**第三版**：做成独立的 AskUserQuestion 工具。

Claude 想提问就必须显式调用它，调用即暂停，没有歧义。

下面这张图能解释为什么第三版明显更稳：

左边（markdown 自由输出）太松，模型格式随意、外层解析脆弱；右边（ExitPlanTool 参数）太死，等到退出计划阶段提问已经太晚；**AskUserQuestion 独立工具落在中间，结构化且随时可调用**。

说白了，既然你就是要 Claude 停下来问一句，那就直接给它一个专门的工具。加个 flag 或者约定一段输出格式，很多时候它一顺手就略过去了。

这个故事让我意识到：**给 agent 设计工具，和给人写 API 不是一回事**。给人用的 API 往往会追求功能齐全，但给 agent 用，重点不是功能堆得多完整，而是让它更容易用对。

### Todo 工具：当初的保护变成枷锁

还有一个更有意思的教训。

早期用 TodoWrite 工具 + 每 5 轮插入提醒让 Claude 记住任务。这个设计在当初是合理的——模型不够强，容易忘事，需要一个"外脑"。

但随着模型变强，这个工具反而成了限制。Todo 提醒让 Claude 认为必须严格遵循计划，无法灵活修改。

**当初加这个工具是因为模型不够强，模型变强之后它反而变成了枷锁。**

这个教训让我反思：很多设计决策都有时效性。值得过段时间回来检查一下，当初加的限制还成不成立。

### 搜索工具的演进

搜索工具也经历过类似的迭代：

最初用 RAG 向量数据库，虽然快但需要索引、不同环境脆弱，最重要的是 Claude 不喜欢用。

改成 Grep 工具让 Claude 自己搜索后，好用很多。

后来又发现一个顺带的好处：Claude 读 Skill 文件，Skill 文件又引用其他文件，模型会递归读取，按需发现信息。这个模式后来被叫做"渐进式披露"。

## 第三幕：实践智慧

### 你的 200K 去哪了？

Claude Code 的 200K 上下文并非全部可用。我用了一段时间才意识到这个问题。

```
200K 总上下文
├── 固定开销 (~15-20K)
│   ├── 系统指令: ~2K
│   ├── 所有启用的 Skill 描述符: ~1-5K
│   ├── MCP Server 工具定义: ~10-20K  ← 最大隐形杀手
│   └── LSP 状态: ~2-5K
│
├── 半固定 (~5-10K)
│   ├── CLAUDE.md: ~2-5K
│   └── Memory: ~1-2K
│
└── 动态可用 (~160-180K)
    ├── 对话历史
    ├── 文件内容
    └── 工具调用结果
```

一个典型 MCP Server（如 GitHub）包含 20-30 个工具定义，每个约 200 tokens，合计 4,000-6,000 tokens。

接 5 个 Server，光这部分固定开销就到了 **25,000 tokens（12.5%）**。

我第一次算出这个数字的时候，真没想到有这么多。在要读大量代码的场景，这 12.5% 真的很关键。

### 上下文的分层治理

理解了这些之后，我开始有意识地分层管理上下文：

```
始终常驻    → CLAUDE.md：项目契约 / 构建命令 / 禁止事项
按路径加载  → rules：语言 / 目录 / 文件类型特定规则
按需加载    → Skills：工作流 / 领域知识
隔离加载    → Subagents：大量探索 / 并行研究
不进上下文  → Hooks：确定性脚本 / 审计 / 阻断
```

**说白了，偶尔用的东西就不要每次都加载进来。**

### Subagent：隔离比并行更重要

关于 Subagent，我一开始也理解错了。

我以为它的价值是"并行"——同时派多个 agent 去干不同的事。但用久了才发现，**它最大的价值是隔离**。

扫代码库、跑测试、做审查这类会产生大量输出的事，塞进主线程很快就把有效上下文挤没了。交给 Subagent 做，主线程只拿一个摘要，干净很多。

子智能体可能跑了 30+ 次工具调用，但整个消息历史直接丢弃。父智能体收到的只是一段摘要文本。

这就好比：**你问"这个项目用什么测试框架"，可能要读 5 个文件，但你只需要一个词："pytest"。**

### HANDOFF.md：一个实用技巧

还有一个技巧我用了很久，觉得特别有用。

开新会话前，先让 Claude 写一份 HANDOFF.md，把当前进度、尝试过什么、哪些走通了、哪些是死路、下一步该做什么写清楚。

下一个 Claude 实例只读这个文件就能接着做，不依赖压缩算法的摘要质量。

这比等着系统自动压缩靠谱多了。压缩算法按"可重新读取"判断，早期的 Tool Output 和文件内容会被优先删掉，**顺带把架构决策和约束理由也一起扔了**。两小时后再改，可能根本不记得两小时前定了什么。

## 结尾：三个阶段

用 Claude Code 这半年，我大概经历了三个阶段：

**第一阶段：探索** —— 当 ChatBot 用，问什么答什么，碰到问题就换工具、加规则。

**第二阶段：掌握** —— 理解了 Agent Loop、工具设计、上下文工程，开始有意识地配置 Skills 和 Hooks。

**第三阶段：治理** —— 关注点从"这个功能怎么用"变成"怎么让 agent 在约束下自己跑起来"。

到了第三阶段，有一个问题挺值得想的：**假如一个任务你说不清楚"什么叫做完"，那大概率也不适合直接扔给 Claude 自主完成。**

验证标准本身都没有，Claude 再聪明也跑不出正确答案。

## 番外：还想聊的

### 真正的长期记忆

现在的 Claude Code 记忆是"会话级"的——每次压缩就像喝断片，醒来只记得摘要。s11 里有个有趣的设计叫"身份重注入"：

```python
if len(messages) <= 3:  # 说明发生了压缩
    messages.insert(0, {"role": "user",
        "content": f"<identity>You are '{name}', role: {role}...</identity>"})
```

智能体也有"失忆症"。

真正的长期记忆应该是什么样？也许是一个独立的知识库，智能体随时可以查。这超出了当前 Claude Code 的范围，但是一个值得探索的方向。

### 六层架构：一个总结框架

最后，如果想系统地理解 Claude Code，可以把它拆成六层：

```
上下文层 → 控制层 → 工具层 → 验证层 → 协作层 → 持久化层
```

只强化其中一层，系统就会失衡。CLAUDE.md 写太长，上下文先污染自己了；工具堆太多了，选择就搞不清楚了；subagents 开得到处都是，状态就漂移了；验证这步跳过了，出了问题根本不知道是哪里挂的。

---

> **"模型就是智能体。我们的工作就是给它工具，然后让开。"**

这些是半年折腾下来的一些总结。如果你也有类似的困惑，希望这篇文章能给你一些输入。
