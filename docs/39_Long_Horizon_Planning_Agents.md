# Plan-and-Act: Improving Planning of Agents for Long-Horizon Tasks

## 层 1：电梯演讲 (30 秒)

**一句话概括**：这篇论文提出将 Planner 和 Executor 分离，并用合成数据微调，在 WebArena-Lite 上达到 57.58% SOTA 成功率，在 WebVoyager 上达到 81.36% 文本最优，解决了 LLM 在长序列任务中规划能力不足的问题。

---

## 层 2：故事摘要 (5 分钟)

### 核心问题

2025 年，Web Agent 面临一个明显的瓶颈：

**"单个 LLM 同时负责规划和执行，在长序列任务中容易迷失。"**

想象这个场景：
```
任务："在 GitHub 上找到最活跃的 contributor，然后 follow 他"

单模型方法：
Step 1: 点击"Contributors"✅
Step 2: 分析贡献排名 ✅
Step 3: ... (忘记最初的目标了)
Step 4: 随便点了个用户 ❌

问题：模型被细节淹没，丢失了全局规划
```

### 核心洞察

UC Berkeley 的 Amir Gholami 团队提出：

**"为什么不让一个模型专门规划，另一个模型专门执行？"**

**"就像公司里 CEO 负责战略，员工负责执行。"**

关键洞察：
1. **规划需要全局视角**（高瞻远瞩）
2. **执行需要关注细节**（脚踏实地）
3. **分开 specialization 比 single model 更好**

### 核心方法

**Plan-and-Act 的工作流程：**

```
用户："Follow the top contributor of this GitHub project"

↓

Planner:
"好的，让我制定一个计划：

Step 1: 导航到'Contributors'页面
Step 2: 识别贡献最多的用户
Step 3: Follow 该用户"

↓

Executor (执行 Step 1):
Action: click("Contributors")
Observation: 页面跳转到 contributors 列表

↓

Planner (更新计划):
"Step 1 完成。继续：

Step 2: 找到贡献数最多的用户
Step 3: Follow 该用户"

↓

Executor (执行 Step 2):
Action: find_element("user[1]")
Observation: "John Doe - 1,234 contributions"

↓

Executor (执行 Step 3):
Action: click("Follow")
Observation: "Successfully followed John Doe"

↓

任务完成！✅
```

### 关键结果

| 基准 | 方法 | 成功率 |
|------|------|--------|
| WebArena-Lite | ReAct | 9.85% |
| WebArena-Lite | LLMCompiler | 14.21% |
| **WebArena-Lite** | **Plan-and-Act** | **57.58%** |
| WebVoyager | SeeAct | 73.2% |
| **WebVoyager** | **Plan-and-Act (text-only)** | **81.36%** |

## 层 3：深度精读

### 开场：一个失败的 Web Agent

**2024 年夏天，UC Berkeley 实验室。**

博士生 Lutfi Erdogan 正在调试一个 Web Agent。

任务很简单：
```
"在 GitHub 上找到这个项目的 top contributor，然后 follow 他"
```

Agent 的执行日志：
```
Step 1: 点击"Insights"标签 ✅
Step 2: 点击"Contributors"链接 ✅
Step 3: 滚动页面查看列表 ✅
Step 4: ... (停住了)
Step 5: 点击了第 3 个用户 ❌
Step 6: 迷路了，开始随机点击 ❌
```

Lutfi 分析了失败原因：

**"模型不是不会点击，而是不知道'为什么要点击'。"**

**"它记得要'找 contributor'，但忘了最终目标是'follow'。"**

**"在长序列中，模型被细节淹没，丢失了全局目标。"**

这时，Amir Gholami 走过来：

**"这不就是人类也会遇到的问题吗？"**

**"所以人类会写 to-do list，会把大任务分解成小步骤。"**

**"为什么不让 LLM 也这样做？"**

这就是 Plan-and-Act 的起点。

### 第一章：为什么长序列任务这么难？

#### 挑战 1：认知负荷

**单模型方法的困境：**
```
LLM 需要同时：
1. 记住最终目标
2. 跟踪当前进度
3. 决定下一步动作
4. 处理页面变化
5. 从错误中恢复

→ 认知负荷过载
```

**心理学类比**：
```
人类工作记忆：7±2 个条目
LLM 的"注意力"：同样有限

当任务超过 10 步，性能显著下降
```

#### 挑战 2：动态环境

**Web 环境的特点：**
- 页面会变化（点击后跳转）
- 元素会消失（弹窗关闭）
- 状态会更新（登录后可见新内容）

**问题**：
```
Planner 制定的计划：
Step 1: 点击 A
Step 2: 点击 B
Step 3: 点击 C

执行完 Step 1 后，页面变了！
Step 2 的元素可能不存在了...

需要动态调整计划！
```

#### 挑战 3：缺乏训练数据

**核心问题**：
```
LLM 没有被训练来做规划！

预训练目标：预测下一个 token
规划需要：分解目标、跟踪进度、调整策略

→ 能力不匹配
```

**数据稀缺**：
```
高质量规划数据 < 1000 条
通用文本数据 > 1 万亿 tokens

→ LLM 更擅长聊天，不擅长规划
```

### 第二章：Plan-and-Act 的设计

#### 架构总览

```
┌─────────────────────────────────────────┐
│             User Query                  │
│ "Follow the top contributor"            │
└─────────────┬───────────────────────────┘
              │
┌─────────────▼───────────────────────────┐
│             Planner                     │
│                                         │
│ 输入：用户目标 + 当前页面状态             │
│ 输出：高层计划 (Step 1, 2, 3...)          │
│                                         │
│ 特点：                                  │
│ - 关注"做什么"(What)                     │
│ - 不关注"怎么做"(How)                    │
│ - 根据反馈调整计划                       │
└─────────────┬───────────────────────────┘
              │ 计划步骤
┌─────────────▼───────────────────────────┐
│             Executor                    │
│                                         │
│ 输入：当前计划步骤 + 页面 HTML           │
│ 输出：具体动作 (click, type, scroll...)  │
│                                         │
│ 特点：                                  │
│ - 专注执行当前步骤                        │
│ - 处理页面细节                           │
│ - 返回执行结果                           │
└─────────────┬───────────────────────────┘
              │ 动作
┌─────────────▼───────────────────────────┐
│            Environment                  │
│            (Web Browser)                │
│                                         │
│ 执行动作 → 页面变化 → 返回新状态          │
└─────────────────────────────────────────┘
```

#### Planner 设计

**输入 Prompt**：
```
你是一个规划助手。你的任务是帮助用户完成 web 导航任务。

当前任务：{user_query}
当前页面：{initial_html_state}

请生成一个逐步计划来实现目标。

计划格式：
## Step 1
[描述]

## Step 2
[描述]

...
```

**输出示例**：
```
## Step 1
导航到"Contributors"页面

## Step 2
识别贡献最多的用户

## Step 3
Follow 该用户
```

**动态调整机制**：
```
每次 Executor 执行后，Planner 会看到：
- 执行结果（成功/失败）
- 新页面状态

然后 Planner 可以：
1. 继续下一步（如果成功）
2. 调整计划（如果失败）
3. 终止任务（如果完成）
```

#### Executor 设计

**输入 Prompt**：
```
你是一个 web 导航助手。你的任务是执行具体的网页操作。

当前目标：{current_plan_step}
当前页面：{current_html}

可用动作：
- click(selector)
- type(selector, text)
- scroll(direction)
- navigate(url)

请生成下一个动作。
```

**输出示例**：
```
Action: click("a[href='/contributors']")
```

**执行流程**：
```
1. 解析动作
2. 在浏览器中执行
3. 等待页面加载
4. 提取新状态
5. 返回结果
```

### 第三章：合成数据生成

#### 动机

**问题**：规划数据极度稀缺

```
人工标注规划数据：
- 成本：$10/条
- 时间：30 分钟/条
- 规模：最多 1000 条

→ 不足以微调大模型
```

**解决**：用 LLM 自动生成合成数据

#### Pipeline 总览

```
┌─────────────────────────────────────────┐
│     阶段 1: 动作轨迹生成                   │
│                                         │
│ Seed Queries → Teacher LLM → 成功轨迹     │
└─────────────┬───────────────────────────┘
              │
┌─────────────▼───────────────────────────┐
│     阶段 2: 计划标注                      │
│                                         │
│ 成功轨迹 → Teacher LLM → 计划 + 轨迹对     │
└─────────────┬───────────────────────────┘
              │
┌─────────────▼───────────────────────────┐
│     阶段 3: 数据增强                      │
│                                         │
│ 种子数据 → 变异 → 多样化样本              │
└─────────────────────────────────────────┘
```

#### 阶段 1：动作轨迹生成

**方法**：使用 WebRL (ICLR 2025) 生成成功轨迹

```
输入：用户查询
输出：成功的动作序列

示例：
Query: "Find the postal code for Sagamore Hill"

轨迹：
1. click("Search")
2. type("SearchBox", "Sagamore Hill Oyster Bay")
3. click("Result[1]")
4. extract("Postal Code")
5. exit("11771")
```

**规模**：
- WebArena-Lite: 1113 条成功轨迹
- WebVoyager: 824 条成功轨迹

#### 阶段 2：计划标注

**方法**：让 Teacher LLM (DeepSeek-R1) 从轨迹反推计划

```
输入：动作轨迹
输出：对应的计划

示例：
轨迹：[click("Contributors"), find("user[1]"), click("Follow")]

计划：
Step 1: 导航到 Contributors 页面
Step 2: 找到排名第一的用户
Step 3: Follow 该用户
```

**关键点**：
- 计划是"目标导向"的（要做什么）
- 轨迹是"动作导向"的（怎么做）
- Planner 学习从目标到动作的映射

#### 阶段 3：数据增强

**方法**：对种子数据进行变异

**变异策略**：

| 策略 | 描述 | 示例 |
|------|------|------|
| Query 改写 | 同义替换 | "Find" → "Locate" |
| 任务泛化 | 抽象化 | "Follow user" → "Interact with user" |
| 任务特化 | 具体化 | "Click" → "Click the red button" |
| 难度调整 | 增加/减少步骤 | 3 步 → 5 步 |

**增强后规模**：
```
原始：1113 条
增强后：~5000 条
```

### 第四章：实验结果

#### 实验 1：WebArena-Lite

**基准介绍**：
- WebArena 的人工验证子集
- 100 个任务，覆盖多种场景
- 评估指标：任务成功率

**结果**：

| 方法 | 基座模型 | 成功率 |
|------|---------|--------|
| GPT-4o | GPT-4o | 32.1% |
| ReAct | GPT-4o | 9.85% |
| LLMCompiler | GPT-4o | 14.21% |
| SeeAct | GPT-4o | 38.5% |
| **Plan-and-Act** | LLaMA-3.3-70B | **57.58%** |

**洞察**：
- 开源模型 + 好的架构 > 闭源模型
- 规划 + 执行分离是关键

#### 实验 2：WebVoyager

**基准介绍**：
- 1000+ 真实 web 任务
- 包含视觉和文本两种输入

**结果**：

| 方法 | 输入 | 成功率 |
|------|------|--------|
| SeeAct | 视觉 + 文本 | 73.2% |
| WebRL | 视觉 + 文本 | 76.5% |
| **Plan-and-Act** | **文本** | **81.36%** |

**洞察**：
- 纯文本 Plan-and-Act 超越多模态方法
- 规划比视觉更重要（在这些任务上）

#### 实验 3：消融实验

**问题**：各个组件的贡献？

| 配置 | WebArena-Lite |
|------|---------------|
| 完整 Plan-and-Act | 57.58% |
| - 合成数据 | 32.1% |
| - Planner 微调 | 28.5% |
| - Executor 微调 | 35.2% |
| - 动态重规划 | 41.3% |

**洞察**：
- 合成数据至关重要（+25%）
- 动态重规划很重要（+16%）

#### 实验 4：规划长度分析

**问题**：Plan-and-Act 能处理多长的任务？

| 规划步数 | 单模型 | Plan-and-Act |
|---------|-------|--------------|
| 1-3 步 | 85% | 92% |
| 4-6 步 | 62% | 78% |
| 7-10 步 | 35% | 65% |
| >10 步 | 12% | 48% |

**洞察**：
- 在长序列任务上优势更明显
- >10 步仍有挑战（未来方向）

### 第五章：局限性与未来

#### 局限性 1：推理延迟

**问题**：
```
每次动作需要：
1. Planner 推理 (~2s)
2. Executor 推理 (~1s)
3. 浏览器执行 (~1s)

总计：~4s/动作

10 步任务 = 40 秒 = 用户可能失去耐心
```

**优化方向**：
- 小模型蒸馏
- 并行推理
- 缓存机制

#### 局限性 2：错误传播

**问题**：
```
Planner 制定错误计划 → Executor 执行失败

示例：
Planner: "点击 Contributors"
但页面上没有这个链接...
Executor: ❌ 失败
```

**解决**：
- Executor 反馈给 Planner
- Planner 调整计划
- 增加"错误恢复"步骤

#### 局限性 3：领域特定

**问题**：
```
在 Web 上训练 → 在 Web 上有效
但其他领域呢？

- 手机 App 导航？
- 桌面软件操作？
- 机器人控制？
```

**未来**：跨领域迁移

---

### 总结

**核心贡献**：
1. Planner + Executor 分离架构
2. 合成数据生成 Pipeline
3. WebArena-Lite 57.58% SOTA
4. WebVoyager 81.36% 文本最优

**关键洞察**：
- 长序列任务需要分离规划和执行
- LLM 不擅长规划，需要微调
- 合成数据是解决数据稀缺的关键

**未来方向**：
- 降低推理延迟
- 跨领域迁移
- 处理>10 步的超长任务
