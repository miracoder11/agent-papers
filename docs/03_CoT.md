# Chain-of-Thought: 解锁大语言模型推理能力的一把钥匙

## 开场：一个数学题的失败

2021年秋天，Google Research的办公室里，Jason Wei盯着屏幕上一个完全错误的答案，感到困惑。

屏幕上是一个简单的数学应用题："Roger有5个网球。他又买了2罐网球，每罐有3个。现在Roger总共有多少个网球？"

GPT-3的回答是："11个。"

答案是对的，但推理过程完全不对。模型似乎只是"猜"对了答案。Jason换了一道类似的题目："食堂有23个苹果。如果他们用了20个做午餐，然后又买了6个。现在食堂有多少个苹果？"

这次模型的回答是："9个。"

又对了！但当Jason继续追问"你是怎么得到这个答案的？"，模型就开始胡说八道了。

"它知道答案，但不知道是怎么得到答案的，" Jason对同事Xuezhi Wang说，"这就像一个学生会背答案，但不会解题。"

Xuezhi点点头："更糟糕的是，当题目变难时，它的准确率急剧下降。在GSM8K数学应用题数据集上，GPT-3只有不到30%的准确率。"

当时是2021年，大语言模型已经展现出惊人的能力——写诗、翻译、编写代码。但在需要多步推理的任务上，它们的表现令人失望。

"为什么？" Jason问，"人类解这种题时，会一步步思考：5 + 2×3 = 5 + 6 = 11。模型为什么不能这样做？"

这个简单的问题，将引领他们做出一项改变AI领域的研究。

## 第一章：研究者的困境

2021年的AI研究领域正处在一个奇怪的境地。

大语言模型（如GPT-3、PaLM）在简单任务上表现出色，但在复杂推理任务上却频频失败。更令人困惑的是，增加模型尺寸似乎无法解决这个问题——即使5400亿参数的PaLM，在数学应用题上的准确率也只有30%左右。

Jason和团队开始深入分析失败案例。他们发现，当模型直接给出答案时，它似乎在"跳过"推理步骤，直接从问题跳跃到答案。这种跳跃在简单问题上可能碰巧正确，但在复杂问题上几乎必然失败。

"传统方法是怎么处理这个问题的？" Xuezhi在一次团队会议上问。

"监督学习，" 有人回答，"用标注了推理步骤的数据训练模型。但这需要大量人工标注，成本很高。"

"而且每个任务都要重新训练，" 另一人补充，"数学题训练的模型无法应用到常识推理上。"

团队陷入了两难：微调方法成本高且不灵活，标准提示方法（Standard Prompting）在推理任务上表现差。

"等等，" Jason突然说，"我们能不能通过提示来让模型展示推理步骤？"

会议室里安静下来。

"你的意思是，" Xuezhi慢慢地说，"不给模型标注数据，而是在提示中展示几个例子，每个例子都包含推理步骤，然后让模型模仿？"

"对，" Jason说，"就像教学生做数学题，不是只告诉他答案，而是展示解题过程。"

这个想法听起来简单，但当时没有人知道它是否真的有效。更重要的是，即使它有效，为什么有效？

## 第二章：试错的旅程

### 第一阶段：最初的尝试

Jason和他的团队开始了实验。他们的第一个想法非常直观：在提示中加入推理步骤。

他们写了一个这样的提示：

```
Q: 罗杰有5个网球。他又买了2罐网球，每罐有3个。现在Roger总共有多少个网球？
A: 罗杰开始有5个球。2罐每罐3个是6个球。5 + 6 = 11。答案是11。

Q: 食堂有23个苹果。如果他们用了20个做午餐，然后又买了6个。现在食堂有多少个苹果？
A:
```

然后让模型补全。

结果出来了。GSM8K数据集上的准确率从27%（标准提示）跃升到58%（LaMDA 137B模型）。

"这不是真的吧，" Xuezhi看着结果说，"仅仅加了几句推理描述，性能就提升了一倍？"

团队开始疯狂实验。他们尝试了不同的推理任务：算术推理、常识推理、符号推理。每个任务上，加入推理步骤都显著提升了性能。

"但为什么？" Jason问，"模型明明训练时没有看过这种推理格式，为什么现在就能用了？"

### 第二阶段：涌现能力的发现

为了理解这个现象，团队做了一个关键实验：在不同尺寸的模型上测试Chain-of-Thought (CoT) prompting。

结果令人震惊：

```
模型尺寸        标准提示    CoT提示
LaMDA 422M     12.1%       9.5%   (下降了！)
LaMDA 2B       15.2%       14.6%  (还是下降)
LaMDA 8B       21.1%       23.1%  (开始提升)
LaMDA 68B      31.5%       41.3%  (显著提升)
LaMDA 137B     34.7%       58.1%  (巨大跃升！)
```

"这太神奇了，" Jason说，"CoT不仅在大模型上有效，而且在大模型上效果非常好。在小模型上，它甚至有害！"

这种现象被称为"涌现能力"（Emergent Ability）——一种能力只在大规模模型上出现，小模型完全不具备。

"这就像，" Xuezhi打比方说，"小孩子无论你怎么教，都无法理解抽象概念。但到了某个年龄，突然就懂了。CoT也是一样——模型需要足够大才能'理解'推理。"

### 第三阶段：控制实验

团队还需要确认：CoT真的起作用了吗？还是其他因素导致的提升？

他们设计了几个控制实验：

**实验1：方程式提示**
```
Q: ...
A: Let x = 网球数, x = 5 + 2×3 = 11
```
结果：几乎无效

**实验2：可变计算（输出点号）**
```
Q: ...
A: ............  (输出与方程等长的点序列)
```
结果：与baseline相当

**实验3：答案后的推理**
```
Q: ...
A: 11. Because 5 + 2×3 = 11  (先给答案再推理)
```
结果：与baseline相当

"关键不是增加了输出长度，也不是让模型展示计算，" Jason总结，"而是推理必须在答案**之前**生成，而且必须以自然语言形式。"

这个发现揭示了CoT的核心机制：通过生成中间推理步骤，模型将复杂问题分解为多个简单子问题，每个子问题都能用模型已有的知识解决。

## 第三章：核心概念 - 大量实例

Chain-of-Thought (CoT)的核心思想非常简单：让模型在给出最终答案之前，先生成中间推理步骤。但要真正理解它的威力，我们需要从多个角度来审视。

### 概念 1：推理链的本质

**【生活类比 1：解数学题】**
想象你在教一个学生做数学题：
- **标准提示**："答案是11"
- **CoT提示**："开始有5个，2罐每罐3个是6个，5+6=11，答案是11"

学生通过看完整的解题过程，学会了**方法**而不是**答案**。

**【生活类比 2：导航指引】**
给别人指路时：
- **标准提示**："往前走"
- **CoT提示**："往前走到红绿灯，左转，直走200米，右手边就是"

CoT提供了**中间步骤**，让导航更清晰可靠。

**【生活类比 3：烹饪食谱】**
教人做饭：
- **标准提示**："炒这道菜"
- **CoT提示**："先热锅，倒油，放葱姜，炒菜，调味"

分解复杂任务为可执行的步骤。

**【代码实例 1：算术推理】**
```
Q: 原来有9台电脑。每天增加5台，4天后有多少台？
A (标准): 29
A (CoT): 每天增加5台，4天增加 5×4=20台。
        原来有9台，现在有 9+20=29台。
        答案是29。
```

**【代码实例 2：常识推理】**
```
Q: 萨米想去有人多的地方，他会去哪里？
   (a) 赛道 (b) 人口稠密地区 (c) 沙漠 (d) 公寓 (e) 路障
A (CoT): 答案必须是人们多的地方。赛道、沙漠、公寓、路障人不多，
        但人口稠密地区人很多。所以答案是。
```

**【代码实例 3：符号推理】**
```
Q: 硬币正面朝上。你翻转硬币两次。硬币还是正面吗？
A (CoT): 硬币被翻转了2次。2是偶数，所以硬币回到原始状态。
        硬币开始是正面，所以现在还是正面。答案是yes。
```

**【代码实例 4：日期推理】**
```
Q: 音乐会原定1943年6月1日，但推迟一天到今天。今天是10天前的日期是多少？
A (CoT): 1943年6月1日的一天后是1943年6月2日，所以今天是1943年6月2日。
        今天之前的10天是1943年5月23日。所以答案是05/23/1943。
```

**【代码实例 5：逻辑推理】**
```
Q: 以下句子是否合理？"Joao Moutinho在NFL比赛中接到了屏幕传球"
A (CoT): Joao Moutinho是足球运动员，不是美式足球运动员。
        屏幕传球是美式足球术语。因此这句话不合理。答案是不。
```

**【对比场景 1：CoT vs 标准提示】**

**标准提示：**
```
Q: 食堂有23个苹果。用了20个做午餐，又买了6个。有多少个？
A: 9 (碰巧对了，但下次可能错)
```

**CoT提示：**
```
Q: 食堂有23个苹果。用了20个做午餐，又买了6个。有多少个？
A: 23 - 20 = 3，3 + 6 = 9，答案是9 (学会了方法，可持续)
```

**【对比场景 2：CoT vs 方程式】**

**CoT（自然语言）：**
```
A: 开始5个，2罐每罐3个是6个，5+6=11
→ 成功率：58%
```

**方程式（数学符号）：**
```
A: Let x = 5 + 2×3 = 11
→ 成功率：12%（几乎无效）
```

**【对比场景 3：推理在前 vs 推理在后】**

**推理在前（CoT）：**
```
A: 5+2×3=5+6=11。答案是11。
→ 成功率：58%
```

**推理在后：**
```
A: 11。因为5+2×3=11。
→ 成功率：14%（无效）
```

**【逐步演化实例】**

**版本 1：直接回答（小模型）**
```
Q: 5 + 2×3 = ?
A: 11 (可能猜的，不知道为什么)
→ 准确率：15%
```

**版本 2：标准提示（中等模型）**
```
Q: 5 + 2×3 = ?
A: 11 (知道答案，但不知道过程)
→ 准确率：27%
```

**版本 3：CoT提示（大模型）**
```
Q: 5 + 2×3 = ?
A: 2×3=6，5+6=11。答案是11。
→ 准确率：58%（飞跃提升！）
```

**版本 4：Zero-shot CoT**
```
Q: 5 + 2×3 = ?
A: 让我们一步步思考。2×3=6，5+6=11。答案是11。
→ 无需示例，只需"让我们一步步思考"
```

### 概念 2：涌现能力

**【什么是涌现？】**

涌现能力是指某些能力只在大规模模型上出现，小模型完全不具备。CoT就是一个典型的涌现能力。

**【涌现的曲线】**

```
性能
  |
  |           ___________ CoT开始起飞
  |          /
  |         /
  |________/
  |____________________ 模型尺寸
      小    中    大
```

**【为什么需要大模型？】**

1. **知识容量**：大模型有更多世界知识
2. **推理能力**：大模型学会了更抽象的推理模式
3. **语言理解**：大模型更好地理解指令和示例
4. **模式匹配**：大模型能识别和应用复杂的推理模式

### 概念 3：推理类型

CoT支持多种类型的推理：

**【类型 1：算术推理】**
```
Q: 原来有9台电脑。每天增加5台，4天后有多少台？
A: 每天增加5台，4天增加 5×4=20台。
   原来有9台，现在有 9+20=29台。
   答案是29。
```

**【类型 2：常识推理】**
```
Q: 梨会在水中下沉吗？
A: 梨的密度约为0.6 g/cm³，小于水。因此梨会浮起来。
   所以答案是否定的。
```

**【类型 3：符号推理】**
```
Q: 硬币正面朝上。你翻转硬币两次。硬币还是正面吗？
A: 硬币被翻转了2次。2是偶数，所以硬币回到原始状态。
   硬币开始是正面，所以现在还是正面。答案是yes。
```

**【类型 4：逻辑推理】**
```
Q: 如果下雨，地面会湿。地面湿了，下雨了吗？
A: 不一定。地面湿可能是因为下雨，但也可能是其他原因
   （如洒水车）。所以不能确定。
```

**【类型 5：多跳推理】**
```
Q: A比B大3岁，B比C大2岁。A比C大多少岁？
A: A比B大3岁，B比C大2岁，所以A比C大 3+2=5岁。
   答案是5。
```

## 第四章：预期 vs 实际

### 你的直觉 vs CoT 的实现

| 维度 | 你的直觉/预期 | CoT 实际实现 | 为什么有差距？ |
|------|--------------|-------------|---------------|
| 何时有效？ | 所有模型都有效 | 只在大模型上有效（>100B） | 推理能力需要足够的模型容量 |
| 什么格式？ | 任何推理格式都行 | 必须是自然语言，推理在前 | 方程式、推理在后都无效 |
| 需要多少示例？ | 越多越好 | 4-8个最优 | 太少学不会，太多干扰 |
| 对简单任务？ | 总是有帮助 | 可能有害 | 简单任务不需要分解 |
| Zero-shot可行吗？ | 不行 | 可以！只需"Let's think step by step" | 模型已经学会了推理模式 |

### 反直觉问题

**问题 1：CoT在小模型上有用吗？**

[先想 1 分钟...]

直觉可能说："应该有用吧？展示推理步骤总比不展示好。"

**实际：** 在小模型上，CoT反而**有害**！

**为什么？**
```
LaMDA 422M: 标准 12.1% → CoT 9.5% (下降！)
LaMDA 2B:   标准 15.2% → CoT 14.6% (还是下降)
LaMDA 8B:   标准 21.1% → CoT 23.1% (开始提升)
```

小模型没有足够的：
- 知识容量来理解推理
- 语言能力来生成推理
- 推理能力来应用推理

**问题 2：方程式推理比自然语言更好吗？**

[先想 1 分钟...]

直觉可能说："方程式更精确，应该更好。"

**实际：** 方程式推理**几乎无效**！

**实验结果：**
```
自然语言CoT:  58% 准确率
方程式CoT:    14% 准确率 (几乎无效)
标准提示:     27% 准确率
```

**为什么？**
- 方程式需要模型理解符号（小模型不会）
- 自然语言更符合训练数据分布
- 模型更擅长自然语言推理

**问题 3：推理在答案前还是后有区别吗？**

[先想 1 分钟...]

直觉可能说："一样吧，反正都有推理。"

**实际：** 推理**必须在答案前**！

**实验结果：**
```
推理在前（CoT）:     58% 准确率
推理在后:            14% 准确率
标准提示:            27% 准确率
```

**为什么？**
- 推理在前：模型通过推理得到答案
- 推理在后：答案已经生成，推理是"事后合理化"
- 模型不是通过推理得到答案，而是先猜答案再编造推理

### 预测-验证循环

**在继续阅读前，预测一下：**

如果让模型用CoT解这个题，它会生成什么样的推理？

**Q:** "食堂有23个苹果。如果他们用了20个做午餐，然后又买了6个。现在食堂有多少个苹果？"

**你的预测：**
```
A: _________________________________
```

[继续阅读看实际输出]

**实际输出：**
```
A: 食堂原来有23个苹果。他们用了20个做午餐，所以剩下 23-20=3 个。
   他们又买了6个，所以现在有 3+6=9 个。
   答案是9。
```

**你的预测和实际有什么不同？**

关键观察：
1. **分解步骤**：将问题分解为两步（先用、后买）
2. **每步计算**：明确计算每一步（23-20=3，3+6=9）
3. **自然语言**：用文字解释，不是方程式
4. **推理在前**：先推理，再给答案

## 第五章：反直觉挑战

### 挑战 1：更多示例 = 更好性能？

**直觉假设：** 更多few-shot示例应该让模型学得更好。

**实验：** 测试不同数量的few-shot示例对性能的影响。

**结果：**
```
0个示例（Zero-shot）:  42%
1个示例:              53%
4个示例:              56%
8个示例:              58% (最优)
16个示例:             55% (下降！)
```

**反直觉发现：** 8个示例最优，16个反而下降！

**为什么？**

1. **示例太少（<4）的问题：**
   - 模型没有充分学习推理模式
   - 容易模仿格式但不懂方法

2. **示例太多（>8）的问题：**
   - **上下文拥挤**：占用太多token
   - **干扰信息**：不同类型的题目可能混淆
   - **过度拟合**：模型可能记住示例而不是学习方法

3. **最优数量（4-8）：**
   - 足够学习推理模式
   - 不会过度干扰
   - 保持在上下文窗口内

**教训：** 在提示工程中，"多"不等于"好"。找到最优数量是关键。

### 挑战 2：CoT在简单任务上有帮助吗？

**直觉假设：** CoT应该对所有推理任务都有帮助。

**实验：** 在不同难度的任务上测试CoT。

**结果：**
```
简单算术（SingleOp）:
  标准提示: 85%
  CoT提示:  84% (下降！)

复杂算术（GSM8K）:
  标准提示: 27%
  CoT提示:  58% (巨大提升！)
```

**反直觉发现：** CoT在简单任务上**有害**！

**为什么？**

1. **简单任务不需要分解**
   - 5 + 3 = ? 直接回答更快
   - 分解反而增加出错机会

2. **复杂任务必须分解**
   - 多步推理需要中间步骤
   - 不分解容易跳步出错

3. **模型容易"过度推理"**
   - 在简单任务上也生成长推理链
   - 增加了出错概率

**教训：** CoT不是万能的。它只在需要多步推理的复杂任务上才有价值。

### 挑战 3：Zero-shot CoT可行吗？

**直觉假设：** Few-shot是必需的，Zero-shot应该不行。

**实验：** 测试只需加一句"Let's think step by step"的Zero-shot CoT。

**结果：**
```
标准Zero-shot:   17%
Zero-shot CoT:   44% (巨大提升！)
Few-shot CoT:    58%
```

**反直觉发现：** 一句话就能触发推理能力！

**为什么？**

1. **模型已经学会了推理模式**
   - 在训练数据中见过大量推理
   - 只需要"触发"而不是"教"

2. **"Let's think step by step"是强指令**
   - 明确告诉模型要一步步思考
   - 模型理解并应用这个指令

3. **Few-shot的边际收益递减**
   - 从Zero-shot到Few-shot提升巨大
   - 但Few-shot内部差异不大

**教训：** 有时候简单的指令比复杂的示例更有效。

## 第六章：关键实验的细节

### 实验 1：GSM8K数学应用题

**任务类型：** 需要多步算术推理的应用题

**CoT vs 基线方法：**

| 模型 | 标准提示 | CoT提示 | 提升 |
|------|---------|---------|------|
| GPT-3 175B | 17.1% | 33.1% | +16% |
| LaMDA 137B | 34.7% | 58.1% | +23.4% |
| **PaLM 540B** | **26.7%** | **58.5%** | **+31.8%** |

**关键发现：** PaLM 540B + CoT达到了SOTA，超过了微调的GPT-3！

**成功案例分析：**

**案例：网球问题**
```
Q: Roger有5个网球。他又买了2罐网球，每罐有3个。
   现在Roger总共有多少个网球？

标准提示（错误）：
A: 11 (碰巧对了，但下次类似问题可能错)

CoT提示（正确）：
A: Roger开始有5个球。2罐每罐3个是6个球。
   5 + 6 = 11。答案是11。
   (学会了方法：5 + 2×3 = 5 + 6 = 11)
```

**关键成功因素：**
1. **问题分解**：将"5 + 2×3"分解为"2×3=6, 5+6=11"
2. **中间步骤**：明确计算每一步
3. **自然语言**：用文字解释，不是方程式
4. **推理在前**：先推理，再给答案

### 实验 2：模型尺寸与CoT效果

**实验设置：** 测试不同尺寸模型的CoT效果

**结果：**

```
模型尺寸        标准提示    CoT提示    CoT提升
LaMDA 422M     12.1%       9.5%      -2.6% (有害)
LaMDA 2B       15.2%       14.6%     -0.6% (有害)
LaMDA 8B       21.1%       23.1%     +2.0% (开始有效)
LaMDA 68B      31.5%       41.3%     +9.8% (显著)
LaMDA 137B     34.7%       58.1%     +23.4% (巨大)
```

**分析：**

1. **小模型（<8B）：** CoT有害或无效
   - 模型没有足够的推理能力
   - 生成推理链反而增加出错机会

2. **中等模型（8-68B）：** CoT开始有效
   - 模型开始理解推理模式
   - 但效果有限

3. **大模型（>100B）：** CoT效果显著
   - 模型有强大的推理能力
   - CoT大幅提升性能

**涌现阈值：** 约100B参数

**教训：** CoT是涌现能力，需要足够大的模型才能有效。

### 实验 3：推理格式的重要性

**实验设置：** 测试不同推理格式的效果

**结果：**

| 格式 | GSM8K准确率 |
|------|------------|
| 标准提示（无推理） | 26.7% |
| 方程式推理 | 14.2% (更差) |
| 推理在后 | 14.7% (更差) |
| **自然语言CoT（推理在前）** | **58.5%** (最佳) |

**关键发现：** 只有自然语言CoT才有效！

**案例对比：**

**格式1：自然语言CoT（有效）**
```
A: 2×3=6，5+6=11。答案是11。
→ 清晰，易于理解
```

**格式2：方程式CoT（无效）**
```
A: Let x = 5 + 2×3 = 11
→ 模型不理解符号语言
```

**格式3：推理在后（无效）**
```
A: 11。因为5+2×3=11。
→ 模型先猜答案，后编造推理
```

**教训：** 推理的格式至关重要。必须是自然语言，且推理必须在答案之前。

## 第七章：与其他方法对比

### CoT vs 其他方法对比

| 维度 | 标准提示 | 微调+CoT | CoT提示 | ReAct | Reflexion |
|------|---------|---------|---------|-------|----------|
| **需要训练？** | ❌ | ✅ | ❌ | ❌ | ❌ |
| **需要标注？** | ❌ | ✅ (昂贵) | ❌ | ❌ | ❌ |
| **泛化能力？** | 高 | 低 | 高 | 中 | 中 |
| **需要大模型？** | 否 | 否 | **是** | 是 | 是 |
| **GSM8K** | 26.7% | 42.0% | **58.5%** | - | - |
| **可解释性？** | 低 | 中 | **高** | 高 | 高 |

### 详细对比

#### 1. CoT vs 标准提示

**相似点：**
- 都通过提示工程实现
- 都不需要训练
- 都使用预训练模型

**不同点：**

| 方面 | 标准提示 | CoT提示 |
|------|---------|---------|
| 推理过程 | 隐藏 | 显式 |
| 性能（复杂任务） | 差 | 好 |
| 性能（简单任务） | 好 | 可能差 |
| 可解释性 | 低 | 高 |
| 模型要求 | 低 | 高（>100B） |

**何时使用标准提示：**
- 简单任务（单步问答）
- 小模型（<100B参数）
- 快速响应需求

**何时使用CoT提示：**
- 复杂推理任务
- 大模型（>100B参数）
- 需要可解释性

#### 2. CoT vs 微调方法

**相似点：**
- 都能提升推理能力
- 都生成推理链

**不同点：**

| 方面 | 微调方法 | CoT提示 |
|------|---------|---------|
| 需要数据 | 大量标注数据 | 几个示例 |
| 需要训练 | 是 | 否 |
| 泛化能力 | 低（任务特定） | 高（跨任务） |
| 成本 | 高 | 低 |
| GSM8K (GPT-3) | 42.0% | 33.1% |

**何时使用微调：**
- 有大量标注数据
- 需要最优性能
- 任务固定，不需要泛化

**何时使用CoT：**
- 数据有限
- 需要跨任务泛化
- 快速原型

#### 3. CoT vs ReAct

**相似点：**
- 都使用推理链
- 都提升推理能力
- 都可解释

**不同点：**

| 方面 | CoT | ReAct |
|------|-----|-------|
| 能行动？ | ❌ | ✅ |
| 外部工具？ | ❌ | ✅ |
| 适用任务 | 纯推理 | 推理+行动 |
| 幻觉问题 | 严重 | 缓解 |

**关系：** ReAct中的"Thought"就是CoT推理。ReAct = CoT + 行动能力

### 局限性分析

#### CoT的核心局限

**1. 需要大模型**
- 小模型上无效甚至有害
- 计算成本高
- 部署困难

**潜在解决方案：**
- 模型蒸馏（大模型教小模型）
- 混合架构（大小模型协作）

**2. 幻觉问题**
- 推理链可能错误但自信
- 无法验证推理正确性
- 事实错误可能传播

**潜在解决方案：**
- 结合外部工具（ReAct）
- 自我验证机制
- 多路径投票

**3. 复杂度限制**
- 对于特别复杂的任务仍然失败
- 推理链长度受限
- 上下文窗口限制

**潜在解决方案：**
- 层次化推理
- 分解为子任务
- 扩展上下文窗口

**4. 任务特异性**
- 某些任务上不如专门方法
- 需要任务特定的提示设计
- 难以自动化

**潜在解决方案：**
- 自动提示优化
- 多任务学习
- 元学习

### 改进方向

#### 1. Self-Consistency
**核心思想：** 多次采样，投票选择最一致的答案

**实现：**
```
生成5条推理链：
推理1: 5+2×3=5+6=11 → 答案11
推理2: 5+2×3=11 → 答案11
推理3: 5+2×3=5+6=11 → 答案11
推理4: 5+2×3=5+6=11 → 答案11
推理5: 5+2×3=5+6=11 → 答案11
→ 投票：11 (5/5一致)
```

**效果：** GSM8K从58.5% → 74.5%

#### 2. Tree of Thoughts
**核心思想：** 探索多条推理路径

**实现：**
```
问题
├─ 路径1: 思考... → 答案A
├─ 路径2: 思考... → 答案B
└─ 路径3: 思考... → 答案C
→ 评估并选择最佳路径
```

**效果：** 在某些任务上超越CoT

#### 3. Auto-CoT
**核心思想：** 自动生成CoT示例

**实现：**
```
1. 从训练集中选择多样性样本
2. 为每个样本生成推理链
3. 构建few-shot提示
```

**效果：** 无需人工设计提示

#### 4. 混合推理
**核心思想：** 结合CoT和其他方法

**实现：**
```
CoT + ReAct: 推理 + 行动
CoT + Reflexion: 推理 + 学习
CoT + 工具使用: 推理 + 外部信息
```

**效果：** 更强大的推理能力

## 第八章：如何应用

理解了CoT的核心机制和实验结果，什么时候应该使用它？

### 场景 1：算术推理

**典型任务：**
- GSM8K、SVAMP这类数学应用题
- 多步计算问题
- 数量关系推理

**为什么需要CoT：**
- 需要多步推理
- 容易跳步出错
- 需要中间验证

**如何设计：**

1. **提供4-8个示例：**
```
Q: 原来有9台电脑。每天增加5台，4天后有多少台？
A: 每天增加5台，4天增加 5×4=20台。
   原来有9台，现在有 9+20=29台。
   答案是29。
```

2. **推理必须在答案之前**
3. **使用自然语言，不是方程式**
4. **确保推理步骤清晰**

### 场景 2：常识推理

**典型任务：**
- CSQA、StrategyQA这类常识问答
- 日常逻辑推理
- 常识判断

**为什么需要CoT：**
- 需要显式推理
- 避免直觉错误
- 提供可解释性

**如何设计：**

1. **强调中间推理：**
```
Q: 梨会在水中下沉吗？
A: 梨的密度约为0.6 g/cm³，小于水。
   因此梨会浮起来。
   所以答案是否定的。
```

2. **注入常识知识**
3. **逐步推导结论**

### 场景 3：符号推理

**典型任务：**
- Coin Flip这类符号操作
- 逻辑推理
- 状态追踪

**为什么需要CoT：**
- 需要追踪状态变化
- 容易出错
- 需要显式推理

**如何设计：**

1. **明确每一步的状态变化：**
```
Q: 硬币正面朝上。你翻转硬币两次。硬币还是正面吗？
A: 硬币被翻转了2次。2是偶数，所以硬币回到原始状态。
   硬币开始是正面，所以现在还是正面。
   答案是yes。
```

2. **追踪关键变量**
3. **明确状态转换规则**

### 场景 4：什么时候不用CoT

**简单问答：**
- "法国的首都是哪里？"
- 模型直接知道答案

**简单计算：**
- "23 + 45 = ?"
- 不需要分解

**快速响应：**
- 实时对话系统
- CoT增加延迟

**小模型：**
- 模型<100B参数
- CoT可能有害

### CoT设计最佳实践

#### 1. Few-shot示例选择

**原则：** 选择代表性、多样性的示例

**好的示例：**
```
Q: [简单加法]
A: [推理步骤]

Q: [乘法]
A: [推理步骤]

Q: [混合运算]
A: [推理步骤]
```

**差的示例：**
```
Q: [都是简单加法]
A: ...
Q: [还是简单加法]
A: ...
Q: [还是简单加法]
A: ...
(缺乏多样性)
```

#### 2. 推理链设计

**原则：** 清晰、完整、自然

**好的推理链：**
```
A: 开始5个，2罐每罐3个是6个，5+6=11。答案是11。
```

**差的推理链：**
```
A: 5+2×3=11。答案是11。 (跳步)
A: 5个球，2罐，3个，算一下是11。 (不清晰)
```

#### 3. Zero-shot CoT

**实现：**
```
Q: [问题]
A: Let's think step by step. [推理]
```

**效果：** 在某些任务上接近Few-shot CoT

#### 4. Self-Consistency

**实现：**
```
for i in range(5):
    answer = model.generate(question)
    answers.append(answer)

final_answer = majority_vote(answers)
```

**效果：** GSM8K从58.5% → 74.5%

### 实战案例：设计一个CoT Agent

**任务：** 数学应用题求解

**步骤 1：设计Few-shot提示**
```
Q: 罗杰有5个网球。他又买了2罐网球，每罐有3个。
   现在Roger总共有多少个网球？
A: Roger开始有5个球。2罐每罐3个是6个球。
   5 + 6 = 11。答案是11。

Q: 食堂有23个苹果。如果他们用了20个做午餐，
   然后又买了6个。现在食堂有多少个苹果？
A: [待完成]
```

**步骤 2：实现CoT生成**
```python
def cot_solve(question):
    prompt = build_few_shot_prompt(question)
    return llm.generate(prompt)
```

**步骤 3：实现Self-Consistency**
```python
def self_consistent_solve(question, n=5):
    answers = []
    for i in range(n):
        answer = cot_solve(question)
        answers.append(answer)
    return majority_vote(answers)
```

**步骤 4：测试和优化**
- 测试不同数量的示例
- 测试Zero-shot vs Few-shot
- 测试Self-Consistency的采样次数
