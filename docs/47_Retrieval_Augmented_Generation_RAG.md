# Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks (RAG)

**论文信息**: Patrick Lewis et al., Facebook AI Research, 2020
**arXiv**: [2005.11401](https://arxiv.org/abs/2005.11401)
**引用数**: 10000+ (截至 2024 年)
**领域**: 检索增强生成、知识密集型任务

---

## 层 1：电梯演讲

**一句话概括**：Facebook AI 在 2020 年提出 RAG（检索增强生成），通过结合参数化记忆（预训练 seq2seq 模型）和非参数化记忆（Wikipedia 向量索引），让语言模型能够检索外部知识来生成更准确、更具体、更可验证的回答，在开放域问答等知识密集型任务上达到 SOTA。

## 层 2：故事摘要

### 背景：2020 年语言模型的知识困境

**时间**：2019-2020 年
**地点**：Facebook AI Research (FAIR)
**人物**：Patrick Lewis 团队

当时的困境：

```
预训练语言模型的局限：
┌─────────────────────────────────────────┐
│ BERT (2018): 110M 参数，知识存在权重里   │
│ GPT-2 (2019): 1.5B 参数，同样的问题      │
│ T5 (2019): 11B 参数，还是同样的问题      │
└─────────────────────────────────────────┘

核心问题：
1. 知识更新困难 - 要添加新知识必须重新训练
2. 无法追溯来源 - 不知道答案从哪里来
3. 容易"幻觉" - 自信地编造错误信息
4. 知识密集型任务表现差 - 需要具体事实时抓瞎
```

**研究者的洞察**

想象一下 Lewis 团队的思考过程：

> "人类是怎么回答知识性问题的？"
>
> "不是靠死记硬背，而是会查资料！"
>
> "比如问'《神曲》分为哪三部分？'"
> "- 专家可能直接回答（参数化记忆）"
> "- 普通人会 Google 一下（非参数化记忆）"
>
> "为什么不让模型也学会查资料呢？"

这就是 RAG 的核心思想。

### 技术路线图

```
┌───────────────────────────────────────────┐
│           RAG 架构总览                     │
├───────────────────────────────────────────┤
│ 输入 x                                     │
│   ↓                                       │
│ ┌─────────────────────────────────────┐   │
│ │  Retriever (检索器)                  │   │
│ │  - 把输入变成向量                    │   │
│ │  - 在 Wikipedia 索引中找最相关的文档  │   │
│ │  - 返回 top-K 文档                    │   │
│ └─────────────────────────────────────┘   │
│   ↓ 检索到的文档 z                        │
│ ┌─────────────────────────────────────┐   │
│ │  Generator (生成器)                  │   │
│ │  - BART (预训练 seq2seq 模型)          │   │
│ │  - 结合输入 x 和文档 z 生成回答        │   │
│ │  - 可微分，端到端训练                │   │
│ └─────────────────────────────────────┘   │
│   ↓                                       │
│ 输出 y                                     │
└───────────────────────────────────────────┘
```

### 核心创新

**1. 两种 RAG 变体**

```
RAG-Sequence:
- 用同一篇文档生成整个回答
- 适合事实性问答
- 边际化：p(y|x) ≈ Σ p(z|x) × p(y|x,z)

RAG-Token:
- 不同 token 可以用不同文档
- 适合需要综合多源信息的任务
- 更灵活但也更复杂
```

**2. 端到端训练**

```
关键洞察：
- 检索器和生成器都可以预训练
- RAG 的创新在于可以联合微调
- 检索器学会"检索对生成有用的文档"
- 生成器学会"利用检索到的文档生成"
```

**3. 无需监督的检索**

```
传统方法：
- 需要标注"应该检索哪篇文档"
- 标注成本极高

RAG 方法：
- 检索的文档作为隐变量
- 通过生成质量反向传播指导检索
- 模型自己学会检索什么
```

### 第一章：研究者的困境

#### 1.1 2020 年的 NLP 格局

**预训练模型的成就**

```
BERT (2018):
- 在 SQuAD 问答超过人类
- 在 GLUE 基准刷新记录
- 但都是 fine-tuning 模式

GPT-2 (2019):
- 展示了惊人的生成能力
- 可以写故事、写诗
- 但事实准确性堪忧

T5 (2019):
- 统一了 NLP 任务框架
- "万事皆可 seq2seq"
- 但知识密集型任务依然困难
```

**知识密集型任务的定义**

```
什么是"知识密集型任务"？

定义：人类无法在没有外部知识源的情况下完成的任务

例子：
✅ 开放域问答："谁写了《战争与和平》？"
✅ 事实核查："地球是平的吗？"
✅ 专业问答："治疗糖尿病的一线药物是什么？"
❌ 情感分析："这部电影很好看"→正面
❌ 语法纠错：修正句子错误
❌ 文本分类：新闻分类

关键区别：
- 知识密集型需要"知道什么"
- 非知识密集型需要"知道怎么做"
```

#### 1.2 现有方法的局限

**方法 1：纯参数化记忆**

```
代表：BERT, GPT, T5

优点：
- 端到端训练
- 推理快速（一次前向传播）
- 可以学习抽象模式

缺点：
- 知识更新需要重新训练
- 无法追溯知识来源
- 容易产生幻觉
- 参数有限，知识容量有限
```

**方法 2：检索 + 抽取**

```
代表：传统 QA 系统

流程：
检索 → 抽取答案片段 → 输出

优点：
- 答案来自可靠来源
- 可以追溯来源
- 知识可以随时更新

缺点：
- 只能抽取，不能生成
- 无法综合多源信息
- 灵活性差
```

**方法 3：记忆网络**

```
代表：Memory Networks, Neural Turing Machines

想法：
- 给模型一个外部记忆
- 可以读写记忆

问题：
- 记忆需要从零学习
- 没有预训练的知识
- 效果不如预期
```

#### 1.3 核心研究问题

团队在论文中提出的核心问题：

> **能否结合参数化和非参数化记忆的优势，同时保持端到端训练的能力？**

具体目标：

```
1. 保留预训练模型的知识
2. 添加可检索的外部知识
3. 端到端微调
4. 适用于任何 seq2seq 任务
5. 可以追溯来源
6. 知识可以更新
```

### 第二章：RAG 的设计

#### 2.1 架构详解

**整体流程**

```
输入："谁写了《神曲》？"
       ↓
┌──────────────────────────────┐
│  Query Encoder (检索器)       │
│  把问题编码成向量             │
│  q(x) = [0.23, -0.45, ...]    │
└──────────────────────────────┘
       ↓
┌──────────────────────────────┐
│  Document Index (文档索引)    │
│  1400 万 Wikipedia 文档向量    │
│  MIPS 搜索找到最相似的         │
└──────────────────────────────┘
       ↓
检索到的文档：
z1: "《神曲》是意大利诗人但丁的史诗作品..."
z2: "但丁·阿利吉耶里（1265-1321）..."
z3: "中世纪文学代表作《神曲》..."
       ↓
┌──────────────────────────────┐
│  Generator (BART)            │
│  结合问题和文档生成回答       │
│  p(y | x, z1, z2, z3)        │
└──────────────────────────────┘
       ↓
输出："《神曲》是意大利诗人但丁·阿利吉耶里创作的长诗..."
```

**RAG-Sequence vs RAG-Token**

这是论文的核心技术创新，需要仔细理解。

```
RAG-Sequence 模型：
思想：同一篇文档负责生成整个序列

流程：
1. 检索 top-K 文档
2. 对每篇文档，计算生成整个序列的概率
3. 加权平均（边际化）

数学公式：
p(y|x) ≈ Σ_z p(z|x) × p(y|x,z)
       = Σ_z p(z|x) × Π_i p(y_i | x, z, y_<i)

直观理解：
- 假设有 K 个"专家"（文档）
- 每个专家给出完整答案的概率
- 最后加权平均
```

```
RAG-Token 模型：
思想：每个 token 可以由不同文档生成

流程：
1. 检索 top-K 文档
2. 对每个 token，在所有文档上边际化
3. 不同 token 可以关注不同文档

数学公式：
p(y|x) = Π_i Σ_z p(z|x) × p(y_i | x, z, y_<i)

直观理解：
- 有 K 个专家
- 每个 token 选择最合适的专家
- 更灵活，但也更复杂
```

**对比示例**

```
问题："比较但丁和莎士比亚的写作风格"

检索到的文档：
z1: 关于但丁的资料
z2: 关于莎士比亚的资料
z3: 关于中世纪文学的资料
z4: 关于文艺复兴的资料
z5: 文学风格分析

RAG-Sequence:
- 选择 z1，生成整个回答（只讲但丁）❌
- 或者选择 z2，生成整个回答（只讲莎士比亚）❌
- 无法综合多个文档

RAG-Token:
- 前半部分用 z1（讲但丁）
- 后半部分用 z2（讲莎士比亚）
- 可以综合多个文档 ✅
```

#### 2.2 组件详解

**检索器 (Retriever)**

```
架构：Dual Encoder (DPR)

Query Encoder:
- 输入：问题 x
- 输出：向量 q(x)

Document Encoder:
- 输入：文档 z
- 输出：向量 d(z)
- 预先编码所有 Wikipedia 文档

相似度：内积 q(x) · d(z)
检索：MIPS (Maximum Inner Product Search)
```

**为什么用 DPR？**

```
DPR (Dense Passage Retriever) 优势：

1. 稠密向量检索
   - 比 BM25 等稀疏检索更准确
   - 可以捕获语义相似性

2. 预训练
   - 在 SQuAD 等 QA 数据上训练
   - 已经学会检索相关文档

3. 高效
   - 可以用 FAISS 加速
   - 毫秒级检索 1400 万文档
```

**生成器 (Generator)**

```
架构：BART-large

BART 特点：
- 双向 encoder
- 自回归 decoder
- 去噪预训练（类似 BERT + GPT）
- 擅长生成任务

为什么选 BART？
- 在 T5 和 BERT 之间折中
- 比 T5 更适合生成
- 比 GPT 更擅长理解上下文
```

#### 2.3 训练策略

**预训练 + 微调**

```
阶段 1：预训练（已完成）
- 检索器：在 DPR 数据上预训练
- 生成器：BART 预训练
- 两者都有丰富的知识

阶段 2：微调（RAG 的贡献）
- 端到端微调
- 检索器和生成器一起更新
- 任务特定的 seq2seq 数据
```

**端到端训练的关键**

```
问题：检索的文档是离散的，怎么反向传播？

解决：把检索文档当作隐变量

训练流程：
1. 对于输入 x，检索 top-K 文档
2. 对每篇文档 z，计算生成 y 的概率
3. 边际化得到最终概率
4. 反向传播同时更新检索器和生成器

数学上：
∂L/∂η = Σ_z ∂L/∂p(z|x) × ∂p(z|x)/∂η  (更新检索器)
∂L/∂θ = Σ_z ∂L/∂p(y|x,z) × ∂p(y|x,z)/∂θ  (更新生成器)
```

**为什么有效？**

```
直观理解：

初始状态：
- 检索器随机检索（但有一定能力）
- 生成器随机生成（但有一定能力）

训练过程：
- 生成器："这篇文档没用，换一个"
- 检索器："好的，我试试别的"
- 反复迭代...

最终状态：
- 检索器学会检索对生成有用的文档
- 生成器学会利用检索到的文档
- 两者默契配合
```

### 第三章：核心概念 - 大量实例

#### 3.1 参数化记忆 vs 非参数化记忆

**生活类比 1：人脑 vs 图书馆**

```
参数化记忆 = 人脑
- 知识存在神经元连接里
- 快速访问
- 容量有限
- 难以更新

非参数化记忆 = 图书馆
- 知识存在书本里
- 访问较慢（需要检索）
- 容量几乎无限
- 容易更新（买新书就行）

RAG = 学者做研究
- 既有自己的知识（参数化）
- 也会查图书馆（非参数化）
- 两者结合最强大
```

**生活类比 2：开卷考试**

```
闭卷考试（纯参数化）：
- 只能靠自己记住的知识
- 记不清就瞎编
- 无法查资料

开卷考试（RAG）：
- 可以查教科书（检索）
- 结合自己的理解（生成）
- 答案更准确、可追溯

RAG 就是让模型参加"开卷考试"
```

**代码示例**

```python
# 参数化记忆（传统语言模型）
class ParametricModel:
    def __init__(self):
        self.weights = load_pretrained_weights()

    def answer(self, question):
        # 所有知识都在权重里
        return generate(self.weights, question)

# 非参数化记忆（RAG）
class RAGModel:
    def __init__(self):
        self.generator = load_bart()
        self.retriever = load_dpr()
        self.index = load_wikipedia_index()

    def answer(self, question):
        # 检索相关文档
        docs = self.retriever.search(question, self.index, top_k=5)
        # 结合文档生成回答
        return generate(self.generator, question, docs)
```

#### 3.2 边际化 (Marginalization)

**生活类比：投资决策**

```
场景：你要投资一支股票

方法 1（RAG-Sequence）：
- 咨询 5 个专家
- 每个专家给出完整的投资建议
- 你综合 5 个建议做决定

方法 2（RAG-Token）：
- 咨询 5 个专家
- 对每个具体问题（买什么、何时买、买多少）
- 选择最擅长该问题的专家意见
- 综合成最终决策
```

**数学示例**

```
假设检索到 2 篇文档 z1, z2
要生成回答 y = "巴黎"

RAG-Sequence:
p("巴黎" | x) = p(z1|x) × p("巴黎" | x, z1)
              + p(z2|x) × p("巴黎" | x, z2)

RAG-Token:
对每个 token 独立边际化
p(y1|x) = Σ_z p(z|x) × p(y1|x,z)
p(y2|x) = Σ_z p(z|x) × p(y2|x,z)
...
```

#### 3.3 检索器训练

**如何学会检索？**

```
监督方法（不可行）：
- 标注每个问题应该检索哪篇文档
- 成本：每问答题需要人工找文档
- 不可扩展

RAG 方法（无监督）：
- 不标注应该检索什么
- 通过生成质量反向传播
- 检索器"无师自通"
```

**训练信号流动**

```
例子：
问题："谁写了《哈姆雷特》？"
正确回答："莎士比亚"

训练流程：
1. 检索器检索文档 z1, z2, ..., z5
2. 生成器尝试生成回答
3. 计算损失（与正确答案的差异）
4. 反向传播：
   - 生成器：学会利用文档生成
   - 检索器：学会检索包含"莎士比亚"的文档

关键：
- 没有显式告诉检索器检索什么
- 通过生成质量隐式指导
```

### 第四章：预期 vs 实际

#### 4.1 直觉预测 vs 实际结果

**预期 1：检索越多越好**

| 你的直觉 | 实际 | 为什么 |
|---------|------|--------|
| top-K 越大越好 | K=5 左右最佳 | 太多噪声干扰生成 |
| 检索整篇 Wikipedia | 只检索段落级 | 细粒度更精确 |
| 多次检索更好 | 一次检索足够 | 多次检索误差累积 |

**预期 2：生成器最重要**

| 你的直觉 | 实际 | 为什么 |
|---------|------|--------|
| 生成器决定一切 | 检索器同样重要 | 垃圾进垃圾出 |
| 用最大的生成器 | 中等大小即可 | 检索弥补了规模 |
| 微调只调生成器 | 两者都要微调 | 联合学习才有效 |

**预期 3：RAG 只适合问答**

| 你的直觉 | 实际 | 为什么 |
|---------|------|--------|
| 只适合 QA 任务 | 适合所有知识密集型任务 | 本质是增强知识 |
| 不适合生成任务 | 生成更 factual, diverse | 有依据地生成 |
| 需要任务特定设计 | 通用 seq2seq 框架 | 预训练的优势 |

#### 4.2 反直觉挑战

**问题：如果检索到错误的文档，模型会被误导吗？**

[先想 1 分钟...]

**直觉**：当然会！垃圾进垃圾出。

**实际**：有一定鲁棒性，但不完美。

```
实验结果：
- 检索到无关文档时，模型有时能忽略
- 但强相关但错误的文档会误导模型
- 这是 RAG 的局限之一

改进方向：
- 更好的检索质量判断
- 生成器学会质疑检索结果
- 多文档交叉验证
```

### 第五章：关键实验

#### 5.1 开放域问答结果

**Natural Questions (谷歌搜索查询)**

```
模型对比（准确率%）：
┌─────────────────────────────────────────┐
│ RAG-Sequence:  45.5 (SOTA)              │
```

#### 6.2 局限性分析

**RAG 的局限**

```
1. 检索质量依赖
   - 检索不到就生成不好
   - 检索错误会误导生成

2. 计算开销
   - 每次生成都要检索
   - 比纯生成慢 2-3 倍

3. 领域限制
   - Wikipedia 覆盖的领域较好
   - 专业领域需要特定索引

4. 长文档理解
   - 检索的是段落级文档
   - 需要跨文档推理时有限
```

#### 6.3 改进方向

```
后续工作（2020 年后）：

1. FiD (Fusion-in-Decoder)
   - 更有效地融合多文档
   - 在 RAG 思路上改进

2. SELF-RAG
   - 学会自我反思是否需要检索
   - 减少不必要检索

3. FLARE
   - 迭代式检索生成
   - 边生成边检索

4. 多模态 RAG
   - 检索图像、表格等
   - 扩展到多模态
```

### 第七章：如何应用

#### 7.1 适用场景

**推荐使用 RAG 的场景**：

```
1. 开放域问答
   - 用户问各种问题
   - 需要准确的事实性回答

2. 知识库问答
   - 公司有文档库
   - 用户查询文档内容

3. 研究助手
   - 检索论文
   - 生成摘要

4. 医疗/法律咨询
   - 需要准确引用来源
   - 不能编造信息

5. 新闻摘要
   - 检索相关新闻
   - 生成综合摘要
```

**不推荐的场景**：

```
1. 创意写作
   - 不需要检索事实
   - 纯生成更好

2. 代码生成
   - 代码不是检索来的
   - 需要逻辑推理

3. 简单分类
   - 不需要生成
   - 直接用 BERT

4. 低延迟场景
   - 检索增加延迟
   - 纯生成更快
```

#### 7.2 快速开始

```python
# 使用 HuggingFace RAG
from transformers import RagTokenizer, RagSequenceForGeneration

# 加载模型
tokenizer = RagTokenizer.from_pretrained("facebook/rag-sequence-nq")
model = RagSequenceForGeneration.from_pretrained("facebook/rag-sequence-nq")

# 推理
question = "谁写了《哈姆雷特》？"
inputs = tokenizer(question, return_tensors="pt")
outputs = model.generate(**inputs)
answer = tokenizer.decode(outputs[0])
print(answer)
```

### 总结

**RAG 的核心贡献**：

1. **架构创新**
   - 首次结合预训练检索和预训练生成
   - 端到端微调框架
   - 两种边际化策略

2. **性能突破**
   - 多个 QA 基准 SOTA
   - 生成更 factual, diverse
   - 知识可以更新

3. **实用价值**
   - 开源代码（HuggingFace）
   - 预训练模型可用
   - 广泛应用场景

**一句话总结**：
> RAG 通过结合参数化记忆（预训练模型）和非参数化记忆（可检索知识），让语言模型能够"开卷考试"，在知识密集型任务上实现更准确、更具体、更可追溯的生成。
