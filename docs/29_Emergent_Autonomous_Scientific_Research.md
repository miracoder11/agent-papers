### 第二章：系统设计——多 LLM 协作架构

#### 核心洞察

**关键问题**：单个 GPT-4 能不能完成所有任务？

**答案**：理论上可以，但有以下问题：
- 上下文窗口有限（8K tokens）
- 单次调用可能"遗忘"某些信息
- 无法同时做多件事

**解决方案**：**多 Agent 协作**

```
┌─────────────────────────────────────────┐
│              Planner (GPT-4)            │
│  • 接收用户指令                          │
│  • 分解任务                             │
│  • 协调其他 Agent                       │
│  • 最终决策                             │
└───────────┬─────────────────────────────┘
            │
    ┌───────┼───────┬───────────┬─────────┐
    │       │       │           │         │
    ▼       ▼       ▼           ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐
│ Docs  │ │Internet│ │Python │ │Experiment│ │...│
│Search │ │Search │ │Exec  │ │Control │ │   │
└───────┘ └───────┘ └───────┘ └───────┘ └───────┘
```

#### Agent 组件详解

**1. Planner (GPT-4)**

**职责**：
- 接收用户的高级指令
- 分解成可执行的子任务
- 协调其他 Agent 执行
- 汇总结果，做出决策

**示例 Prompt**：
```
你是一个科研助手。你的任务是帮助设计并执行化学实验。

你可以使用以下工具：
- GOOGLE: 搜索互联网
- PYTHON: 执行代码计算
- EXPERIMENT: 控制实验设备
- DOCS: 查阅文档

用户指令："设计并执行 Suzuki 反应"

请逐步规划任务...
```

**2. Docs Searcher**

**职责**：
- 查阅技术文档
- 检索 API 使用说明
- 理解硬件接口

**技术实现**：
```python
# 使用向量搜索快速定位文档片段

    # 将 query 和文档片段都转换为 embedding

    return relevant_docs
```

**3. Internet Searcher**

**职责**：
- Google 搜索最新文献
- 获取实时信息
- 补充训练数据之外的知识

**示例**：
```
PLANNER: "搜索最新的 Suzuki 反应催化剂"
INTERNET: 返回 Google 搜索结果
PLANNER: "根据搜索结果，选择 Pd(PPh3)4 作为催化剂"
```

**职责**：
- 执行数值计算
- 处理实验数据
- 生成图表

**示例**：
```python

    return reagent1_mol / reagent2_mol

```

**职责**：
- 控制液体处理仪器
- 执行温度控制
- 操作搅拌器等设备

```

对于复杂任务，**分层规划是必需的**。

#### 发现 3：温度参数的影响

团队测试了不同温度设置：

| 温度 | 代码正确率 | 创新性 |
|------|-----------|--------|
| 0.0 | 95% | 低 |
| 0.5 | 85% | 中 |
| 0.8 | 60% | 高 |
| 1.0 | 40% | 很高 |

**反直觉**：

对于科学实验，**低温度（确定性）更重要**。

论文中提到：
> "The high-temperature parameter for text generation results in volatility... For scientific tasks, we recommend temperature < 0.2"

### 第五章：安全讨论

#### 潜在风险

论文专门用一节讨论了安全问题：

**风险 1：双用途研究**

Agent 可能被用于：
- 设计有益分子（药物）
- 设计有害分子（毒剂、生物武器）

**风险 2：自主性带来的失控**

如果 Agent 能够：
- 自我改进代码
- 访问更多工具
- 与其他 Agent 协作

**会不会出现"AI 科学家"失控的情况？**

**风险 3：知识传播**

Agent 的设计方法如果开源：
- 好的方面：加速科研
- 坏的方面：降低作恶门槛

#### 建议的安全措施

团队提出了具体措施：

**1. API 访问控制**
```python
# 白名单机制
ALLOWED_TOOLS = ['GOOGLE', 'PYTHON', 'DOCS']
FORBIDDEN_TOOLS = ['DARK_WEB', 'WEAPON_DESIGN']
```

**2. 输出审查**
```python
# 审查 Agent 的输出
def safety_check(output):
    if contains_dangerous_content(output):
        return BLOCK
    return ALLOW
```

**3. 人类监督**
```
关键决策点 → 人类确认 → 继续执行
```

**4. 日志记录**
```
所有操作 → 详细日志 → 可追溯
```

### 第七章：局限性

#### 局限性 1：依赖训练数据

Agent 的知识受限于 GPT-4 的训练数据：
- 2023 年 4 月之后的研究不知道
- 专业领域知识可能不准确
- 需要互联网搜索补充

#### 局限性 2：无法进行创新性思考

Agent 能**组合已有知识**，但难以**创造全新理论**。

**示例**：
- 能设计已知反应的新组合 ✅
- 能发现全新的反应机理 ❌

#### 局限性 3：长序列任务仍有挑战

超过 10 步的任务容易"迷失"：
- 忘记早期步骤
- 重复执行某些操作
- 需要人类干预

#### 局限性 4：安全风险

如前所述，存在双用途风险。
